# Логирование Serilog, ILogger, структурированное логирование

## Структурированное логирование

`Структурированное логирование` - подход к логированию, когда лог записывается не как строка, а как набор именованных свойств, которые впоследствии можно анализировать и фильтровать.

Обычное текстовое логирование выглядит следующим образом: "User 42 logged in at 2025-09-01 10:00:00"

А в структурированном логировании та же запись будет выглядеть следующим образом:

```json
{
  "Timestamp": "2025-09-01T10:00:00Z",
  "Level": "Information",
  "MessageTemplate": "User {UserId} logged in at {Time}",
  "RenderedMessage": "User 42 logged in at 2025-09-01 10:00:00",
  "Properties": {
    "UserId": 42,
    "Time": "2025-09-01T10:00:00Z"
  }
}
```

Преимущества структурированных логов очевидны, это проще читать и анализировать программными средствами, проще осуществлять поиск по ключевым словам, а сами логи становятся не просто строками в текстовых файлах, а данными, которые можно анализировать.

> Ключевая идея состоит в том, чтобы писать не 
> `Log.Information($"User {userId} logged in");`
> А писать следующим образом
> `Log.Information("User {UserId} logged in", userId);`
> Таким образом UserId становится отдельным полем,
> по которому можно осуществить поиск

## Serilog - царь структурированного логирования

`Serilog` - самая популярная библиотека структурированного логирования для .Net, её NuGet пакет имеет свыше 2.5 миллиардов установок.

К преимуществам `Serilog` относится: структурированное логирование из коробки, приёмники различных данных встроенные в библиотеку, обогатители с широкой экосистемой, гибкая фильтрация, возможность форматировать данные в `Json` из коробки.

`Serilog` выступает провайдером логирования, доступ к которому ты так же осуществляешь через интерфейс `ILogger`.

## Установка Serilog

```bash
dotnet add Serilog.AspNetCore
```

Данный пакет включает в себя:

- Базовый `Serilog`
- Интеграция ASP.NET: `builder.Services.AddSerilog();`
- Поддержка Request Logging

Базовая настройка в `Program.cs`:

```c#
using Serilog;

// 1. Создаём логгер ДО построения хоста
Log.Logger = new LoggerConfiguration()
    .WriteTo.Console()
    .CreateLogger();

try
{
    Log.Information("Starting web application");

    var builder = WebApplication.CreateBuilder(args);

    // 2. Регистрируем Serilog как провайдер логирования
    builder.Services.AddSerilog();

    var app = builder.Build();

    app.MapGet("/", () => "Hello World!");
    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application terminated unexpectedly");
}
finally
{
    Log.CloseAndFlush();
}
```

> После подключения Serilog 
> можно удалить секцию "Logging" 
> из appsettings.json — она больше не нужна

## Ключевые понятия

`Sink` - приёмники, это место, куда `Serilog` отправляет логи

- Console - вывод в консольку входит в пакет: `Serilog.AspNetCore`
- File - Запись в файл с поддержкой ротации. Пакет: `Serilog.Sinks.File`
- Seq - Отправка в ПО `Seq`, позволяющее визуализировать логи. Пакет: `Serilog.Sinks.Seq`
- Elasticsearch - Интеграция с Elasticsearch. Пакет: `Serilog.Sinks.Elasticsearch`
- ApplicationInsights - Интеграция с Azure. Пакет: `Serilog.Sinks.ApplicationInsights`

Пример с ротацией:

```c#
Log.Logger = new LoggerConfiguration()
    .WriteTo.Console()                      // Sink: консоль
    .WriteTo.File("logs/app-.log",          // Sink: файл с ротацией
        rollingInterval: RollingInterval.Day)
    .CreateLogger();
```

`Enrichers` - обогатители, позволяют автоматически добавлять к лог-событию дополнительную информацию.

- `FromLogContext` - добавляет свойства через `ILogger.BeginScope`, входит в пакет `Serilog`
- `WithEnvironmentName` - добавляет окружение в поля логов. Пакет: `Serilog.Enrichers.Environment`
- `WithMachineName` - добавляет имя машины. Пакет: `Serilog.Enrichers.Environment`
- `WithThreadId` - Id потока, где произошло лог-событие. Пакет: `Serilog.Enrichers.Thread`

```c#
Log.Logger = new LoggerConfiguration()
    .Enrich.FromLogContext()
    .Enrich.WithEnvironmentName()
    .Enrich.WithMachineName()
    .WriteTo.Console()
    .CreateLogger();
```

## Интеграция `ILogger<T>` и `IDiagnosticContext`

После вызова `builder.Services.AddSerilog()` все стандартные `ILogger<T>` автоматически будет использовать `Serilog` как провайдер. 

Вот как `Serilog` встроится в Контроллер. 

```c#
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    private readonly ILogger<UsersController> _logger;

    public UsersController(ILogger<UsersController> logger)
    {
        _logger = logger;
    }

    [HttpGet("{id}")]
    public IActionResult GetUser(int id)
    {
        // Структурированный лог — id станет отдельным свойством
        _logger.LogInformation("Getting user {UserId}", id);
        
        // ...
    }
}
```

Использование `IDiagnosticContext` позволяет добавлять свойства к логу завершения `HTTP` запроса

```c#
public class HomeController : Controller
{
    private readonly IDiagnosticContext _diagnosticContext;

    public HomeController(IDiagnosticContext diagnosticContext)
    {
        _diagnosticContext = diagnosticContext;
    }

    public IActionResult Index()
    {
        // Это свойство появится в логе завершения запроса
        _diagnosticContext.Set("CatalogLoadTime", 1423);
        return View();
    }
}
```

## Request Logging - логирование HTTP-запросов

`Serilog` предоставляет Middleware для ASP.NET, позволяющее свернуть все лог-события в рамках обработки одного HTTP запроса в единое событие с методом, путём, статусом и временем выполнения.

```c#
var app = builder.Build();

app.UseSerilogRequestLogging(); // <-- до app.MapControllers()

app.MapControllers();
app.Run();
```

Важно: `UseSerilogRequestLogging()` должна быть до обработчиков (`MVC`, `Minimal API`), но после `middleware`, которые не нужно логировать (например, статические файлы)

Настройка `Request Logging`

```c#
app.UseSerilogRequestLogging(options =>
{
    // Свой шаблон сообщения
    options.MessageTemplate = "Handled {RequestPath}";

    // Уровень логирования (по умолчанию Information)
    options.GetLevel = (httpContext, elapsed, ex) => LogEventLevel.Debug;

    // Добавление своих свойств
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        diagnosticContext.Set("RequestHost", httpContext.Request.Host.Value);
        diagnosticContext.Set("UserAgent", httpContext.Request.Headers["User-Agent"]);
    };
});
```

## Пример Serilog

**Структура проекта**

```bash
MyApi/
├── Program.cs
├── appsettings.json
└── MyApi.csproj
```

**Program.cs**

```c#
using Serilog;

// ═══════════════════════════════════════════════════════════════════════════
// 1. Настройка Serilog до построения хоста
// ═══════════════════════════════════════════════════════════════════════════
// Логгер создаётся ДО WebApplication.CreateBuilder(), чтобы перехватывать
// ошибки, возникающие во время настройки приложения (например, ошибки
// чтения appsettings.json, проблемы с портами и т.п.).
Log.Logger = new LoggerConfiguration()
    // Уровень логирования: Information и выше (Debug не пишется)
    .MinimumLevel.Information()
    // Для Microsoft-логов (ASP.NET Core) ставим уровень Warning,
    // чтобы не засорять логи информационными сообщениями от фреймворка.
    .MinimumLevel.Override("Microsoft", LogEventLevel.Warning)
    // Enricher: позволяет добавлять свойства через BeginScope()
    // и делает их доступными в каждом лог-событии.
    .Enrich.FromLogContext()
    // Enricher: добавляет свойство EnvironmentName (Development/Production)
    // к каждому лог-событию.
    .Enrich.WithEnvironmentName()
    // Sink: вывод в консоль с кастомным шаблоном.
    // {Timestamp:HH:mm:ss} — время, {Level:u3} — уровень (INF/WRN/ERR),
    // {Message:lj} — сообщение, {Properties:j} — все дополнительные свойства в JSON,
    // {Exception} — стек-трейс при ошибке.
    .WriteTo.Console(outputTemplate:
        "[{Timestamp:HH:mm:ss} {Level:u3}] {Message:lj} {Properties:j}{NewLine}{Exception}")
    .CreateLogger(); // Создаём экземпляр логгера

try
{
    // Первое лог-событие — сигнал о старте приложения.
    // Если приложение падает на старте, это событие будет видно.
    Log.Information("Starting web application");

    // ═══════════════════════════════════════════════════════════════════════════
    // 2. Создание и настройка хоста ASP.NET Core
    // ═══════════════════════════════════════════════════════════════════════════
    var builder = WebApplication.CreateBuilder(args);

    // Регистрируем Serilog как провайдер логирования вместо стандартного ILogger.
    // Теперь все вызовы ILogger<T> из кода будут обрабатываться Serilog.
    builder.Services.AddSerilog();

    // Добавляем контроллеры и OpenAPI (для документации).
    builder.Services.AddControllers();
    builder.Services.AddOpenApi();

    // Строим приложение.
    var app = builder.Build();

    // ═══════════════════════════════════════════════════════════════════════════
    // 3. Настройка Middleware
    // ═══════════════════════════════════════════════════════════════════════════
    // Этот middleware должен быть ДО MapControllers().
    // Он сворачивает весь HTTP-запрос в ОДНО лог-событие:
    // метод, путь, статус, длительность выполнения.
    app.UseSerilogRequestLogging();

    // В режиме разработки отображаем OpenAPI-документ.
    if (app.Environment.IsDevelopment())
    {
        app.MapOpenApi();
    }

    // Подключаем контроллеры.
    app.MapControllers();

    // Запускаем приложение.
    app.Run();
}
catch (Exception ex)
{
    // Если произошло НЕОЖИДАННОЕ исключение (не обработанное внутри приложения),
    // логируем его как Fatal — такие логи обычно вызывают алерты.
    Log.Fatal(ex, "Application terminated unexpectedly");
}
finally
{
    // Принудительно закрываем логгер и сбрасываем буферы.
    // Без этого последние логи могут быть потеряны при аварийном завершении.
    Log.CloseAndFlush();
}
```

**appsettings.json**

```json
{
  "AllowedHosts": "*"
}
```

**Вывод**

```bash
[10:15:23 INF] Starting web application
[10:15:24 INF] Now listening on: http://localhost:5000
[10:15:24 INF] Application started. Press Ctrl+C to shut down.
[10:15:24 INF] Hosting environment: Development
[10:15:30 INF] HTTP GET /api/users/5 responded 200 in 45.23 ms
```

## Конфигурация `Serilog` через `appsettings.json`

В малых проектах сконфигурировать через код - удобно, но большие проекты выводят конфигурацию в `appsettings.json`

**Подключение конфигурации из `appsettings.json`**

```c#
using Serilog;

// 1. Создаём логгер ДО построения хоста
Log.Logger = new LoggerConfiguration()
    .ReadFrom.Configuration(builder.Configuration)
    .CreateLogger();
```

**Секция `Serilog` в `appsettings.json`**

```c#
{
  "Serilog": {
    "MinimumLevel": {
      "Default": "Information",
      "Override": {
        "Microsoft": "Warning",
        "Microsoft.AspNetCore": "Warning",
        "System": "Warning"
      }
    },
    "Enrich": [ "FromLogContext", "WithEnvironmentName", "WithMachineName" ],
    "WriteTo": [
      {
        "Name": "Console",
        "Args": {
          "outputTemplate": "[{Timestamp:HH:mm:ss} {Level:u3}] {Message:lj} {Properties:j}{NewLine}{Exception}"
        }
      }
    ]
  },
  "AllowedHosts": "*"
}
```

Что здесь происходит:
- `MinimumLevel.Default` — глобальный минимальный уровень.
- `MinimumLevel.Override` — переопределение для конкретных пространств имён (чтобы не захламлять логи сообщениями от `ASP.NET Core`).
- `Enrich` — список обогатителей (используются по имени).
- `WriteTo` — список приёмников с параметрами.

**Важно**: Чтобы работало чтение из `appsettings.json`, нужно добавить пакет `Serilog.Settings.Configuration`:

```bash
dotnet add package Serilog.Settings.Configuration
```

**Пример `Program.cs` с конфигурацией:**

```c#
using Serilog;

var builder = WebApplication.CreateBuilder(args);

// 1. Настройка логгера из appsettings.json
Log.Logger = new LoggerConfiguration()
    .ReadFrom.Configuration(builder.Configuration) // читает секцию "Serilog"
    .CreateLogger();

builder.Host.UseSerilog(); // альтернативный способ регистрации

builder.Services.AddControllers();
var app = builder.Build();

app.UseSerilogRequestLogging();
app.MapControllers();

try
{
    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application terminated unexpectedly");
}
finally
{
    Log.CloseAndFlush();
}
```

## Уровни логирования

* `Verbose` - Самый подробный, для отладки
* `Debug` - Отладочная информация, полезная для разработчика
* `Information` - Общая информация о том, что там происходит вообще
* `Warning` - Предупреждение, что что-то идёт не так и могут быть ошибки
* `Error` - Требующая внимания ошибка
* `Fatal` - Критическая ошибка и приложение падает

Уровни логирования для разных компонент системы можно переопределить:

```json
{
  "Serilog": {
    "MinimumLevel": {
      "Default": "Information",
      "Override": {
        "Microsoft": "Warning",
        "Microsoft.AspNetCore": "Warning",
        "System.Net.Http.HttpClient": "Warning",
        "MyApp.Infrastructure": "Debug"
      }
    }
  }
}
```

## Файловый вывод с ротацией

**Установка**

```bash
dotnet add package Serilog.Sinks.File
```

**Конфигурация**

```json
{
  "Serilog": {
    "WriteTo": [
      {
        "Name": "Console"
      },
      {
        "Name": "File",
        "Args": {
          "path": "logs/app-.log",
          "rollingInterval": "Day",
          "outputTemplate": "[{Timestamp:yyyy-MM-dd HH:mm:ss} {Level:u3}] {Message:lj} {Properties:j}{NewLine}{Exception}",
          "retainedFileCountLimit": 31,
          "fileSizeLimitBytes": 10485760
        }
      }
    ]
  }
}
```

Параметры файлового `sink`:

- Параметр - Описание
- `path` -  Шаблон пути (с - для ротации)
- `rollingInterval` -  Day, Hour, Month, Year
- `retainedFileCountLimit` -  Максимальное количество файлов (старые удаляются)
- `fileSizeLimitBytes` -  Лимит размера файла до ротации (по умолчанию 1 ГБ)

## `WithCorrelationId` - связывание логов

**Корреляционный Id** это уникальный идентификатор, добавляемый в лог в рамках обработки одного запроса, что позволяет связывать логи из разных компонент системы обработки одного запроса воедино, что позволяет прослеживать процесс обработки.

**Установка** 

```bash
dotnet add package Serilog.Enrichers.CorrelationId
```

**Настройка `appsettings.json`**

```json
{
  "Serilog": {
    "Enrich": [ "FromLogContext", "WithCorrelationId", "WithEnvironmentName" ]
  }
}
```

**Использование в `Program.cs`**

```c#
app.UseSerilogRequestLogging(options =>
{
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        // Если заголовок X-Correlation-ID есть, используем его,
        // иначе генерируем новый (это делает сам enricher)
        var correlationId = httpContext.Request.Headers["X-Correlation-ID"].FirstOrDefault();
        diagnosticContext.Set("CorrelationId", correlationId);
    };
});
```

Теперь в каждом лог-событии будет поле `CorrelationId`, по которому можно фильтровать логи одного запроса.

## BeginScope() - добавление контекстных свойств

`BeginScope()` позволяет добавить свойства ко всем логам в рамках определённого блока кода (например для обработки HTTP запроса)

```c#
public async Task<IActionResult> GetUser(int id)
{
    // Добавляем UserId ко всем логам внутри этого блока
    using (_logger.BeginScope(new { UserId = id }))
    {
        _logger.LogInformation("Getting user");
        var user = await _userService.GetUserAsync(id);
        _logger.LogInformation("User retrieved");
        return Ok(user);
    }
}
```

В логах появится свойство UserId у всех событий внутри блока.

**Добавление нескольких свойств**

```c#
using (_logger.BeginScope(new Dictionary<string, object>
{
    ["RequestId"] = requestId,
    ["TransactionId"] = transactionId
}))
{
    _logger.LogInformation("Processing order");
    // все логи внутри увидят RequestId и TransactionId
}
```

## Кастомизация вывода

Положим, вы не используете что-то стандартное для сбора логов, а реализовали что-то своё, вместо `ELK` стека, который жрёт ресурсы, а что-то проще и оптимальнее, а стандартного коннектора(`Sink`) для этого дела нет, тогда вы можете написать собственный, кастомный `Sink`. Всё что нужно - реализовать `ILogEventSink`

```c#
public class CustomSink : ILogEventSink
{
    public void Emit(LogEvent logEvent)
    {
        // Отправить куда угодно: в HTTP-эндпоинт, в базу, в очередь, в консоль
        Console.WriteLine($"Custom: {logEvent.RenderMessage()}");
    }
}

// Регистрация в коде
Log.Logger = new LoggerConfiguration()
    .WriteTo.Sink(new CustomSink())
    .CreateLogger();
```

## Тестирование логирования

Да, иногда даже логи приходится тестировать, поэтому кратенько посмотрим, как это делается

```c#
using Microsoft.Extensions.Logging;
using Moq;
using Xunit;

public class UserServiceTests
{
    [Fact]
    public void GetUser_WhenUserNotFound_LogsWarningWithUserId()
    {
        // ═══════════════════════════════════════════════════════════════════
        // 1. ARRANGE — подготовка
        // ═══════════════════════════════════════════════════════════════════

        // Создаём мок для ILogger<UserService>.
        // Мы будем проверять, что метод Log был вызван с правильными аргументами.
        var loggerMock = new Mock<ILogger<UserService>>();

        // Создаём экземпляр сервиса, передавая ему мок логгера.
        // UserService — это тестируемый класс, который внутри вызывает _logger.LogWarning.
        var service = new UserService(loggerMock.Object);

        // ID пользователя, который точно не существует в нашем тестовом контексте.
        int nonExistentUserId = 999;

        // ═══════════════════════════════════════════════════════════════════
        // 2. ACT — выполнение тестируемого действия
        // ═══════════════════════════════════════════════════════════════════

        // Вызываем метод, который должен залогировать предупреждение.
        // Предположим, что GetUser выбрасывает исключение или возвращает null,
        // но мы тестируем именно логирование, а не результат.
        var result = service.GetUser(nonExistentUserId);

        // ═══════════════════════════════════════════════════════════════════
        // 3. ASSERT — проверка, что логгер был вызван правильно
        // ═══════════════════════════════════════════════════════════════════

        // Проверяем, что метод Log был вызван ровно один раз.
        // Мы проверяем все параметры: уровень, сообщение, аргументы.
        loggerMock.Verify(
            x => x.Log(
                // Уровень логирования должен быть Warning
                LogLevel.Warning,

                // EventId — обычно не важен, используем It.IsAny<EventId>()
                It.IsAny<EventId>(),

                // Сообщение — мы проверяем, что оно содержит текст "User not found" и ID пользователя.
                // Используем It.Is<It.IsAnyType> для проверки состояния объекта состояния.
                It.Is<It.IsAnyType>((state, type) =>
                    state.ToString().Contains("User not found") &&
                    state.ToString().Contains(nonExistentUserId.ToString())
                ),

                // Исключение — в данном случае null, так как мы логируем предупреждение, а не ошибку.
                It.IsAny<Exception>(),

                // Функция форматирования — обычно стандартная, используем It.IsAny<Func<...>>.
                It.IsAny<Func<It.IsAnyType, Exception, string>>()
            ),
            // Проверяем, что вызов был ровно один раз.
            Times.Once
        );
    }
```

**Альтернатива `Serilog.Sinks.InMemory`**

Установим: `dotnet add package Serilog.Sinks.InMemory`

И сам пример:

```c#
using Serilog;
using Serilog.Events;
using Serilog.Sinks.InMemory;
using Xunit;

public class UserServiceLoggingTests
{
    [Fact]
    public void GetUser_WhenUserNotFound_LogsWarningWithUserId_InMemorySink()
    {
        // ═══════════════════════════════════════════════════════════════════
        // 1. ARRANGE — подготовка
        // ═══════════════════════════════════════════════════════════════════

        // Создаём InMemorySink — он будет хранить все лог-события в памяти.
        var sink = new InMemorySink();

        // Настраиваем Serilog так, чтобы все события записывались в наш sink.
        // Мы не используем глобальный Log.Logger, а создаём локальный логгер для теста.
        var logger = new LoggerConfiguration()
            .MinimumLevel.Information()       // Устанавливаем минимальный уровень
            .WriteTo.Sink(sink)               // Отправляем все события в InMemorySink
            .CreateLogger();

        // Создаём тестируемый сервис, передавая ему наш локальный логгер.
        // Важно: сервис должен принимать ILogger (стандартный интерфейс Microsoft),
        // но мы передаём Serilog-логгер, который реализует этот интерфейс.
        var service = new UserService(logger);

        // ID пользователя, который не существует.
        int nonExistentUserId = 999;

        // ═══════════════════════════════════════════════════════════════════
        // 2. ACT — выполнение
        // ═══════════════════════════════════════════════════════════════════

        // Вызываем метод, который должен залогировать предупреждение.
        var result = service.GetUser(nonExistentUserId);

        // ═══════════════════════════════════════════════════════════════════
        // 3. ASSERT — проверка записанных событий
        // ═══════════════════════════════════════════════════════════════════

        // Получаем все записанные лог-события из sink.
        var logEvents = sink.LogEvents;

        // Проверяем, что было записано ровно одно событие.
        Assert.Single(logEvents);

        // Берём это событие.
        var logEvent = logEvents.First();

        // Проверяем уровень логирования — должен быть Warning.
        Assert.Equal(LogEventLevel.Warning, logEvent.Level);

        // Проверяем, что сообщение содержит ожидаемый текст.
        // RenderMessage() возвращает готовое сообщение с подставленными значениями.
        var renderedMessage = logEvent.RenderMessage();
        Assert.Contains("User not found", renderedMessage);
        Assert.Contains(nonExistentUserId.ToString(), renderedMessage);

        // Проверяем, что структурированное свойство UserId присутствует и равно ожидаемому.
        // Properties — это словарь, где ключ — имя свойства, значение — ScalarValue.
        Assert.True(logEvent.Properties.ContainsKey("UserId"));
        var userIdProperty = logEvent.Properties["UserId"] as ScalarValue;
        Assert.NotNull(userIdProperty);
        Assert.Equal(nonExistentUserId, userIdProperty.Value);
    }
}
```

## Практические советы

**Структура логов**
- Минимум `Timestamp`, `Level`, `MessageTemplate`, `Properties`
- Для HTTP обязательные поля: `RequestPath`, `Method`, `StatusCode`, `Duration`
- Обязательные поля для деловых процессов: `UserId`, `OrderId`, `TransactionId`
- Обязательное поле для распределённых систем: `CorrelationId`

**Рекомендация по уровням для продакшена:**
- `MinimumLevel.Information` (глобально).
- `MinimumLevel.Override("Microsoft", "Warning")` (для ASP.NET Core).
- `MinimumLevel.Override("MyApp", "Debug")` (для своего кода — если нужно больше деталей).

**Ротация и хранение**
- Файловые логи хранить 30 дней.
- Настроить автоматическую очистку старых файлов (retainedFileCountLimit).
- Для высоконагруженных систем использовать асинхронную отправку (WriteTo.Async).
