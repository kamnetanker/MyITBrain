# Паттерны проектирования

В 1994 году вышла книга «Design Patterns: Elements of Reusable Object-Oriented Software» — её написали Эрих Гамма, Ричард Хелм, Ральф Джонсон и Джон Влиссидес. Четверо авторов, которых в народе прозвали «Бандой четырёх» (Gang of Four, GoF)

Они собрали 23 паттерна — типовых решений для часто встречающихся проблем в ООП-дизайне.

Что такое паттерн: Это не готовый код, который можно скопировать, а описание решения — схема, идея, подход. Это как рецепт: ингредиенты и последовательность действий известны, но каждый повар готовит по-своему.

## Критика и почему к паттернам нужно относиться с осторожностью
Главная проблема: Книгу GoF многие читают как сборник рецептов, а не как сборник идей. В результате разработчики начинают пихать паттерны везде, где только можно и где нельзя.

Основные претензии к GoF:
- Паттерны привязаны к C++ — книга писалась под возможности C++ 90-х. В современных языках многие проблемы решаются проще.
- «Подгоняем проблему под решение» — разработчики ищут, куда бы применить паттерн, вместо того чтобы искать простое решение.
- Спекулятивный дизайн — когда ты создаёшь гибкость, которая никому не нужна, «на всякий случай».
- Догматизм — паттерны начинают использовать как религиозные догмы, а не как инструменты.

Цитата из критики:

«Design Patterns — это корм для безмозглых программистов, неспособных к самостоятельному мышлению, которые будут производить код, столь же посредственный, как и паттерны, которые они используют»

Жёстко, но доля правды есть. Паттерны — это инструмент, а не цель.

## Сценарии применения паттернов

Необходимо сразу определиться, что решать несуществующие проблемы - путь в никуда, проблемы необходимо решать по мере их поступления.

1. Ты уже столкнулся с проблемой, которую решает паттерн - используй паттерн
2. Ты думаешь, что в будущем возникнет проблема - **НЕ используй паттерн**
3. Простое решение из 5 строк закроет задачу здесь и сейчас - **НЕ используй паттерн**
4. Код разрастается, спагеттифицируется, а паттерн реально причешет это безобразие - используй паттерн
5. "Так все делают" - **НЕ используй паттерн**

KISS и YAGNI — твои лучшие друзья. Паттерн, который усложняет код вместо того чтобы упрощать — антипаттерн в данном контексте.

## Классификация паттернов GoF

Всего GoF предложили 23 паттерна, разделённые на 3 группы:

- Порождающие - позволяют гибко создавать объекты. (`Singleton`, `Factory`, `Builder`, `Abstract Factory`, `Prototype`)
- Структурные - позволяют гибко управлять связями между объектами (`Adapter`, `Decorator`, `Facade`, `Proxy`, `Composite`, `Bridge`, `Flyweight`)
- Поведенческие - общение объектов друг с другом (`Observer`, `Strategy`, `Command`, `Template Method`, `Iterator`, `State`, `Visitor`, `Mediator`, `Memento`, `Chain of Responsibility`, `Interpreter`)

Мы разберём самые полезные паттерны, которые реально пригодятся в повседневной разработке. Остальные — либо устарели, либо используются в специфических задачах.

## Порождающие паттерны

1. Factory Method (Фабричный метод)

    Проблема категории Г: у тебя есть код, который должен создавать объекты, но заранее неизвестно, какой именно класс потребуется создать

    Решение: Создаётся метод, который возвращает объект нужного типа. Подклассы переопределяют этот метод и возвращают свои объекты, требуемого типа.

    ```c#
    // ❌ Плохо: жёсткая привязка к конкретному классу
    класс Отчёт {
        метод Создать() {
            документ = новый PDFДокумент()  // что если завтра понадобится Word?
            документ.Сформировать()
        }
    }

    // ✅ Хорошо: фабричный метод
    абстрактный класс Отчёт {
        абстрактный метод СоздатьДокумент()
        
        метод Сформировать() {
            документ = СоздатьДокумент()
            документ.Собрать()
            документ.Сохранить()
        }
    }

    класс PDFОтчёт : Отчёт {
        метод СоздатьДокумент() {
            возврат новый PDFДокумент()
        }
    }

    класс WordОтчёт : Отчёт {
        метод СоздатьДокумент() {
            возврат новый WordДокумент()
        }
    }
    ```

    Когда использовать: Большая иерархия объектов и неизвестно, какой будет создан в рантайме.

    Когда не использовать: у тебя один тип объектов, нет иерархии наследования. Фабрика не нужна.

2. Builder (Строитель)

    Проблема категории Г: Объект сложный, с кучей параметров и конструктор превращается в простыню 15-ти аргументов.

    Решение: объект-строитель, который собирает пошагово объект.

    ```c#
    // ❌ Плохо: конструктор-монстр
    класс Отчёт {
        конструктор(заголовок, автор, дата, списокСтраниц, цветФона, шрифт, размерШрифта, ...) {
            // 20 параметров
        }
    }

    // ✅ Хорошо: строитель
    класс Отчёт {
        поле заголовок
        поле автор
        поле дата
        поле списокСтраниц
        // ...
    }

    класс СтроительОтчёта {
        поле отчёт = новый Отчёт()
        
        метод СЗаголовком(текст) {
            отчёт.заголовок = текст
            возврат этот
        }
        метод САвтором(имя) {
            отчёт.автор = имя
            возврат этот
        }
        метод СДатой(дата) {
            отчёт.дата = дата
            возврат этот
        }
        метод Построить() {
            возврат отчёт
        }
    }

    // Использование:
    отчёт = новый СтроительОтчёта()
        .СЗаголовком("Годовой отчёт")
        .САвтором("Иванов")
        .СДатой(сегодня)
        .Построить()
    ```

    Когда использовать: Когда у объекта много параметров, и некоторые из них опциональны.

    Когда не использовать: Если у объекта 2-3 параметра — просто используй конструктор или new { ... }.

3. Singleton (Одиночка) - антипаттерн!

    Проблема: Нужно, чтобы у класса был только один экземпляр во всей программе.

    Решение: Делаешь конструктор приватным, а доступ к единственному экземпляру — через статический метод.

    Почему это антипаттерн:
    - Нарушает SRP — класс отвечает и за свою логику, и за управление своим экземпляром
    - Создаёт глобальное состояние — трудно тестировать, трудно отлаживать
    - Порождает скрытые зависимости — код, использующий синглтон, не говорит об этом явно
    - В многопоточной среде требует осторожной синхронизации

    ```c#
    // ❌ Плохо: синглтон
    класс Логгер {
        частный статический Логгер экземпляр
        
        частный Логгер() { }
        
        статический метод ПолучитьЭкземпляр() {
            если (экземпляр == null) экземпляр = новый Логгер()
            возврат экземпляр
        }
        
        метод Записать(сообщение) { ... }
    }

    // ✅ Хорошо: просто передавай зависимость через конструктор
    класс Логгер {
        метод Записать(сообщение) { ... }
    }

    класс Сервис {
        частный Логгер логгер
        
        конструктор(Логгер л) {
            логгер = л
        }
    }
    ```

    Когда использовать Singleton: Почти никогда. Если очень надо — используй DI-контейнер, который управляет временем жизни объектов (SingleInstance).

## Структурные паттерны

1. Adapter (Адаптер)

    Проблема: У тебя есть класс с одним интерфейсом, а код ожидает другой интерфейс. Как подружить несовместимое?

    Решение: Создаёшь прослойку-адаптер, которая преобразует один интерфейс в другой.

    ```c#
    // Есть старая система
    класс СтараяСистема {
        метод ОтправитьДанные(строка данные) { ... }
    }

    // Новый код ожидает такой интерфейс
    интерфейс IНовыйОтправитель {
        метод Отправить(объект данные)
    }

    // Адаптер
    класс Адаптер : IНовыйОтправитель {
        частный СтараяСистема старая
        
        конструктор(СтараяСистема с) {
            старая = с
        }
        
        метод Отправить(объект данные) {
            строка json = Сериализовать(данные)
            старая.ОтправитьДанные(json)
        }
    }
    ```

    Когда использовать: Когда ты не можешь изменить старый код, но должен его использовать в новом контексте.

2. Decorator (Декоратор)

    Проблема: Нужно добавлять функциональность объекту динамически, но наследование создаёт слишком много классов.

    Решение: Оборачиваешь объект в другой объект, который добавляет новое поведение.

    ```c#
    интерфейс IКофе {
        метод Стоимость()
        метод Описание()
    }

    класс Эспрессо : IКофе {
        метод Стоимость() { возврат 100 }
        метод Описание() { возврат "Эспрессо" }
    }

    // Декоратор
    абстрактный класс ДекораторКофе : IКофе {
        защищённый IКофе кофе
        
        конструктор(IКофе к) { кофе = к }
        
        метод Стоимость() { возврат кофе.Стоимость() }
        метод Описание() { возврат кофе.Описание() }
    }

    класс СМолоком : ДекораторКофе {
        конструктор(IКофе к) : база(к) { }
        
        метод Стоимость() { возврат база.Стоимость() + 30 }
        метод Описание() { возврат база.Описание() + ", с молоком" }
    }

    класс ССахаром : ДекораторКофе {
        конструктор(IКофе к) : база(к) { }
        
        метод Стоимость() { возврат база.Стоимость() + 10 }
        метод Описание() { возврат база.Описание() + ", с сахаром" }
    }

    // Использование:
    кофе = новый ССахаром(новый СМолоком(новый Эспрессо()))
    // Итог: Эспрессо, с молоком, с сахаром — 140 рублей
    ```

    Когда использовать: Когда нужно гибко добавлять функциональность без создания кучи подклассов.

    Альтернатива: В C# можно использовать extension methods для простых случаев.

3. Facade (Фасад)

    Facade — один из самых простых и полезных структурных паттернов. Добавь его в раздел структурных паттернов после Adapter/Decorator. Пример:

    ```c#
    // Сложный API
    класс Видеоконвертер {
        метод ЗагрузитьФайл(путь)
        метод Декодировать()
        метод ПрименитьФильтр()
        метод Сжать()
        метод Сохранить(путь)
    }

    // Упрощаем через фасад
    класс ФасадКонвертации {
        метод Конвертировать(вход, выход) {
            конвертер = новый Видеоконвертер()
            конвертер.ЗагрузитьФайл(вход)
            конвертер.Декодировать()
            конвертер.ПрименитьФильтр()
            конвертер.Сжать()
            конвертер.Сохранить(выход)
        }
    }
    ```

4. Proxy - для ленивой загрузки или кэширования

    ```c#
    интерфейс IДанные { метод Получить() }

    класс РеальныеДанные : IДанные {
        метод Получить() { /* тяжёлый запрос */ }
    }

    класс ПроксиДанные : IДанные {
        частный РеальныеДанные реальные
        частный кэш
        
        метод Получить() {
            если (кэш == null) {
                реальные = новый РеальныеДанные()
                кэш = реальные.Получить()
            }
            возврат кэш
        }
    }
    ```

## Поведенческие паттерны

1. Strategy (Стратегия)

    Проблема: У тебя есть алгоритм, который должен вести себя по-разному в зависимости от ситуации.

    Решение: Выносишь алгоритм в отдельный класс — стратегию, и передаёшь его в основной класс.

    ```c#
    интерфейс IСпособОплаты {
        метод Оплатить(сумма)
    }

    класс Карта : IСпособОплаты {
        метод Оплатить(сумма) { /* списать с карты */ }
    }

    класс Наличные : IСпособОплаты {
        метод Оплатить(сумма) { /* принять наличные */ }
    }

    класс Криптовалюта : IСпособОплаты {
        метод Оплатить(сумма) { /* перевести крипту */ }
    }

    класс Касса {
        частный IСпособОплаты способ
        
        конструктор(IСпособОплаты с) { способ = с }
        
        метод ПровестиПлатёж(сумма) {
            способ.Оплатить(сумма)
        }
    }

    // Использование:
    касса = новый Касса(новый Карта())
    касса.ПровестиПлатёж(1000)
    ```

    Когда использовать: Когда у тебя есть несколько вариантов поведения, которые могут меняться в рантайме.

2. Observer (Наблюдатель)

    Проблема: Когда один объект меняет состояние, другие объекты должны узнать об этом и отреагировать.

    Решение: Создаёшь механизм подписки: наблюдатели подписываются на события, а объект оповещает всех подписчиков.

    ```c#
    // Событие — это встроенный observer в C#
    класс Магазин {
        событие Действие<Товар> ТоварПоявился
        
        метод ДобавитьТовар(товар) {
            // добавить товар
            ТоварПоявился?.Вызвать(товар)  // оповестить всех
        }
    }

    класс Покупатель {
        метод Купить(товар) {
            // купить товар
        }
    }

    // Использование:
    магазин = новый Магазин()
    покупатель = новый Покупатель()

    магазин.ТоварПоявился += покупатель.Купить  // подписка
    магазин.ДобавитьТовар(новый Товар("Хлеб"))  // покупатель узнает
    ```

    Когда использовать: Когда у тебя есть связь «один-ко-многим» и объекты должны реагировать на изменения друг друга.

    В C# это уже встроено — события и делегаты делают observer тривиальным.

3. Template Method (Шаблонный метод)

    Проблема: У тебя есть алгоритм с фиксированной последовательностью шагов, но некоторые шаги могут отличаться в разных реализациях.

    Решение: Базовый класс определяет скелет алгоритма, а подклассы переопределяют отдельные шаги.

    ```c#
    абстрактный класс Напиток {
        // Шаблонный метод — фиксирует последовательность
        метод Приготовить() {
                НагретьВоду()
                Заварить()
                НалитьВЧашку()
                ДобавитьДобавки()
        }
        
        метод НагретьВоду() { /* общая реализация */ }
        метод НалитьВЧашку() { /* общая реализация */ }
        
        абстрактный метод Заварить()
        абстрактный метод ДобавитьДобавки()
    }

    класс Чай : Напиток {
        метод Заварить() { засыпатьЧай() }
        метод ДобавитьДобавки() { добавитьСахар() }
    }

    класс Кофе : Напиток {
        метод Заварить() { сваритьЗерна() }
        метод ДобавитьДобавки() { добавитьМолоко() }
    }
    ```

    Когда использовать: Когда у тебя есть алгоритм с общими и вариативными частями.

## Паттерны, которые почти не нужны в современном C#
Паттерн	- Почему не нужен:  
- Singleton -	Антипаттерн. Используй DI-контейнер
- Abstract Factory -	Заменяется DI-контейнерами и фабриками с делегатами
- Prototype -	Заменяется клонированием через MemberwiseClone или сериализацией, но редко нужно
- Visitor	Заменяется pattern matching и switch expressions в C# 8+
- Iterator -	В C# есть foreach, IEnumerable, LINQ
- Command -	Заменяется делегатами и лямбдами
- Mediator -	Полезен, но есть готовые библиотеки (MediatR)
- Flyweight -	Актуален только для экстремальной оптимизации памяти
- Interpreter -	Почти никогда не нужен — используй готовые парсеры
- Bridge -	Часто избыточен — используй композицию напрямую
