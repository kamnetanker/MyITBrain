# Архитектурные паттерны

Данная тема, как и ряд предыдущих не касается C# напрямую, но требуется для качественной разработки исходя из принципов архитектурного проектирования.  

Архитектурные паттерны - способ организации систем на макроуровне. Паттерны проектировани отвечают на вопрос "Как организовать классы и методы?", архитектурные же паттерны отвечают на вопрос "Как организовать систему?".

Архитектурные паттерны отвечают на следующие вопросы:

- Как организовать систему послойно;
- Как слои должны взаимодействовать между собой;
- Как данные и потоки управления взаимодействуют внутри системы.

Архитектурный паттерн - не догматическое учение, а способ организации типовых систем, отработанный множеством команд разработки и принятый в коммерческой индустрии как стандарт де-факто. В реальных системах редко происходит четкое использование одного паттерна. Чаще всего применяется несколько паттернов, которые решают свои задачи в рамках системы.

Архитектурные паттерны можно разделить исходя из решаемой задачи:

- Структурные - организуют код по слоям/модулям: `Layered Architecture`, `Hexagonal (Ports&Afapters)`, `Onion`
- Поведенческие - как данные и события движутся: `CQRS`, `Event Sourcing`
- UI-ориентированные - ориентированы на взаимодействие с пользователем: `MVC`, `MVVM`, `MVP`

## MVC - Модель-Вид-Контроллер

`MVC` - один из ранних паттернов, принятых на вооружение веб-сервисами состоит из 3 основных понятий:

- `Model` - модель данных, схемы данных, бизнес-логика. Понятие не имеет о том, как данные будут отображены или кто будет управлять процессом модификации данных.
- `View` - отображение данных контроллеру, не знает о Model напрямую, по сути, содержит интерфейс взаимодействия (не путать с понятием интерфейс, интерфейс взаимодействия - совокупность открытых полей и методов, через которые осуществляется взаимодействие)
- `Controller` - принимает запросы и организует взаимодейтсвие между `view` и `model`

Поток данных и управлени:
Пользователь -> Controller (Обрабатывает запрос) -> Model (Обновляет данные) -> Controller (Выбирает View) -> View (отображает данные)

Пример псевдокодом:

```c#
// Model — данные и бизнес-логика
класс Пользователь {
    поле Имя
    поле Email
    метод Сохранить() { /* запись в БД */ }
}

// Controller — обрабатывает запросы
класс КонтроллерПользователей {
    метод ПоказатьПрофиль(идентификатор) {
        пользователь = База.НайтиПользователя(идентификатор)
        возврат Представление("Профиль", пользователь)
    }
    метод ОбновитьEmail(идентификатор, новыйEmail) {
        пользователь = База.НайтиПользователя(идентификатор)
        пользователь.Email = новыйEmail
        пользователь.Сохранить()
        возврат Перенаправление("Профиль")
    }
}

// View — шаблон для отображения
<h1>Профиль: @Model.Имя</h1>
<p>Email: @Model.Email</p>
```

Каждый запрос обрабатывается одним потоком, архитектура MVC не предполагает параллелизма в рамках обработки одного запроса. Асинхронными могут быть операции ввода-вывода. Сам MVC не требует многопоточности.

Если бизнес-логика не слишком сложна и предполагается веб-приложение или какой-то сетевой сервис, то данный архитектурный паттерн вполне подходит.

Сложная логика превращает контроллер в сложно поддерживаемое разросшееся безобразие, а определённые фронтенд фреймворки смещают акцент в сторону mvvm паттерна.

## MVVM - Model-View-ViewModel

`MVVM` - эволюция MVC для современных UI приложений с двусторонней привязкой данных. Разработан Microsoft для WPF, позже кочевал в MAUI, Avalonia используется во фронтенде (Knockout, Vue);

Состоит из:

- Model - Данные, бизнес-логика (эвкивалент MVC)
- View - интерфейс (XAML, HTML), не содержит бизнес-логики, привязан к ViewModel
- ViewModel - посредник между View и Model. Содержит состояние и команды для View. Реализует `INotifyPropertyChanged`, чтобы уведомлять `View` об изменениях.

Поток:
View-> привязка -> ViewModel -> команда -> Model -> событие -> ViewModel -> уведомление -> View

```c#
// Model
класс Пользователь {
    поле Имя
    поле Email
}

// ViewModel
класс ViewModelПрофиля : INotifyPropertyChanged {
    частный Пользователь пользователь
    
    свойство Имя {
        получить { возврат пользователь.Имя }
        установить { 
            пользователь.Имя = значение
            Уведомить(nameof(Имя))
        }
    }
    
    команда СохранитьКоманда { ... }
    
    событие PropertyChanged
}

// View (XAML)
<TextBox Text="{Binding Имя}" />
<Button Command="{Binding СохранитьКоманда}" />
```

Суть: Есть слой `View` - за него отвечает главный поток исполнения. Он отрисовывает интерфейс и регистрирует интеракции пользователя с приложением. До начала своей работы по взаимодействию с пользователем создаётся поток обработки команд пользователя - слой `ViewModel`, `ViewModel` ожидает получения команд от `View` и увдеомляет `View`, когда команды исполнены и можно обновить отображение. `Model` - бизнес-логика, может асинхронно выполнятся в потоках `ViewModel`.

Не рекомендуется пытаться использовать данный паттерн для  Web-API, поскольку там MVC показывает себя значительно лучше, однако через WebHook (или асинхронный сетевые запросы) можно реализовать двойной слой архитектуры, где MVC в целом выступает бекграундом для ViewModel-Model, а View - веб страница на компьютере пользователя.

Консольные приложения так же не требуют MVVM, это крайне излишне.

## Layered Architecture - слоёный пирог

Приложение делится на горизонтальные слои, где каждый слой отвечает за свой уровень абстракции:

- `Presentation (UI)`: Отображение данных, ввод пользователя
- `Business (BLL)`: Бизнес-логика, правила
- `Data Access (DAL)`: Доступ к БД, внешним API
- `Infrastructure (опционально)`: Логирование, кэширование, конфигурация

Каждый слой может взаимодействовать только с нижележащим (при строгой архитектуре)

```c#
// Data Access Layer
класс РепозиторийПользователей {
    метод Найти(идентификатор) { /* SQL-запрос */ }
}

// Business Layer
класс СервисПользователей {
    частный РепозиторийПользователей репозиторий
    
    метод ПолучитьПрофиль(идентификатор) {
        пользователь = репозиторий.Найти(идентификатор)
        // бизнес-логика (валидация, преобразование)
        возврат пользователь
    }
}

// Presentation Layer
класс КонтроллерПользователей {
    частный СервисПользователей сервис
    
    метод Профиль(идентификатор) {
        пользователь = сервис.ПолучитьПрофиль(идентификатор)
        возврат Представление(пользователь)
    }
}
```

Потоки

Зависит от слоя:

- Presentation (UI) — обычно однопоточный (поток UI) или запросный (веб).
- Business и Data Access — могут использовать многопоточность (параллельная обработка данных, асинхронные вызовы).

Но сама слоистая архитектура не диктует многопоточность — ты решаешь, где и когда её применять.

Используется, как основа для организации систем и часто используется совместно с другими паттернами.

## CQRS (Command Query Responsibility Segregation)

Суть: Разделение команд и запросов на уровне архитектуры.

- Команда - Изменяет состояние (создание, обновление, удаление). Не возвращает данные (кроме идентификатора).
- Запрос - возвращает данные, ничего не меняет.

В простом варианте одной БД - просто разделение интерфейсов взаимодействия. В сложном (более 1 БД) - команды пишут новые данные в одну БД, а запросы читают из реплики, что позволяет оптимизировать нагрузку.

```c#
// Запрос
класс ЗапросПолучитьПользователя {
    поле идентификатор
}

класс ОбработчикЗапроса {
    метод Обработать(запрос) {
        возврат БД.НайтиПользователя(запрос.идентификатор)
    }
}

// Команда
класс КомандаОбновитьEmail {
    поле идентификатор
    поле новыйEmail
}

класс ОбработчикКоманды {
    метод Обработать(команда) {
        пользователь = БД.Найти(команда.идентификатор)
        пользователь.Email = команда.новыйEmail
        БД.Сохранить(пользователь)
    }
}
```

Многопоточность может применятся, но обычно используется однопоточный процесс, возможна асинхронность.

Используется в микросервисах и при масштабировании через разделение чтения и записи.

Личное мнение: бесполезный паттерн, рождает слишком много служебного бесполезного кода.

## Event Sourcing - источник события

Исходное состояние посредством цепочки событий приводит к текущему состоянию. Можно использовать для работы без хранения текущего состояния, а лишь вычисления его (например для валидации в финансовых системах), а можно хранить состояние и цепочку действий, приведших к текущему моменту (для реализации отката).

```c#
// События
событие СчётСоздан { ИдентификаторСчёта, ИмяВладельца }
событие ДеньгиЗачислены { ИдентификаторСчёта, Сумма }
событие ДеньгиСписаны { ИдентификаторСчёта, Сумма }

// Храним только события (в БД событий)
// Текущий баланс вычисляем воспроизведением:
баланс = 0
для каждого события в порядке поступления:
    если событие == ДеньгиЗачислены -> баланс += Сумма
    если событие == ДеньгиСписаны -> баланс -= Сумма
```

Обычно паттерн однопоточный (для полной безопасности).

Позволяет осуществлять аудит событий, откатывать события, валидировать данные.

Не применяй, если тебе не нужно: делать финансовую систему, не нужен редактор с возможностью отката, не делаешь систему безопасности с аудитом событий.

## Hexagonal Architecture (Ports&Adapters)

Суть: бизнес-логика изолирована от внешнего мира, имеет лишь интерфейсы, через которые взаимодействует, реализации могут быть разные, а бизнес логика зависит только от интерфейса, по сути Dependency Inversion.
