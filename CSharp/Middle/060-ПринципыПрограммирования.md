# Принципы программирования (в дополнение к SOLID)

## SOLID

SOLID - 5 принципов проектирования программ, введённых Робертом Мартином. Цель - код должен быть легко поддерживаемым, легко расширяемым и легко понимаемым. 

Принципы SOLID это не истины выбитые на скрижалях и имеют множество позиций критики, когда слепое следование этим принципам приводило к тому, что система была неподдерживаема. Часть из этих примеров связана с неправильным пониманием SOLID, часть с тем, что SOLID применялся там, где ему не место. Об этом в подразделе "Критика".

### S - Single Responsibility (Единственная ответственность)

У класса только одна ответственная зона - его состояние, определяемое областью функционирования.

Класс, ответственный за данные пользователя не должен содержать поле под файловый поток, потому что это не его зона ответственности, как и не должен содержать методов напрямую работающих с файловым вводом-выводом.

```text
// ❌ ПЛОХО: Один класс делает всё
класс Пользователь{
    метод СохранитьНаДиск(аргумент путь); // Сохраняет на диск
    метод ЗагрузитьСДиска(аргумент путь); // Читает с диска
    метод СоздатьПользователя(аргумент данные); // Создание экземпляра пользователя
    метод ИзменитьДанные(аргумент данные); // Меняет данные пользователя
}

/*
В вышеприведённом примере, чтение и запись с диска и на диск должны быть реализованы в классе, работающем с диском, создание из данных экземпляров (парсинг, десериализация) это задача фабрики, возврат нового экземпляра из данных это тоже задача фабрики. Единственное, что должен делать класс пользователя - работать с собственными данными, на основе аргументов. 
*/

// ✅ ХОРОШО: Каждый класс отвечает за одно
класс ФайловыйЧитун{ // Единственная ответственность - работа с файлами
    метод ПрочитатьСДискаТекст(аргумент путь);
    метод ПрочитатьСДискаБинарные(аргумент путь);
}
класс ФабрикаОбъектов{ // Единственная ответственность - сериализация/парсинг/десериализация объектов
    метод СохранитьОбъект(аргумент путь, аргумент объект);
    метод ЗагрузитьОбъект(аргумент путь, ...); // Здесь может быть много реализаций, опустим нюансы
    метод СоздатьОбъект(аргумент данныеДляОбъекта);
}
класс Пользователь{ // Единственная ответственность - данные о пользователе
    метод ДелатьЧто-тоСДаннымиПользователя(аргумент Данные);
}
```

Когда нарушаешь: изменение логики сохранения заставляет перекомпилировать весь класс с генерацией и печатью → риск сломать всё сразу.

### O - Open/Closed класс открыт к расширению, закрыт к изменению.

Суть принципа в том, чтобы подходить к проектированию таким образом, чтобы базовая реализация методов не нарушала логики работы его родительских классов. Т.е. обобщение(апкаст) до родительского класса и алгоритмы, работающие с родительским классом, должны корректно работать с дочерними классами, дочерние классы должны расширять функциональность базового, но не изменять её.

Часто этот принцип, в том числе в популярном изложении, путают к закрытости для изменения функциональности того, что уже написано, однако это невозможно, программа меняется и эволюционирует и код может меняться, это нормально. Плодить сотни наследников под изменение принципа работы - нелогично, спагеттифицирует код, приводит к проблемам с пониманием логики работы.

Здесь разработчик находится в очень сложном положении, выход из которого только через опыт в проектировании и рефакторинг, когда опыт не вывез поставленную задачу.

С одной стороны изменение алгоритма базового класса может привести к нарушению функционирования дочерних и такие изменения необходимо планировать и документировать заранее, определяя план действий. Обратная ситуация, изменение функционирования наследников через механизм наследования, может сломать алгоритмы, работающие через обобщения, что тоже проблема. Когда возникают такие проблемы - вопрос к архитектуре приложения и как образуется логика наследования, возможно, решением является реализация интерфейсов, а не наследование, тем не менее каждый отдельный случай уникален и должен рассматриваться отдельно.

Пример кода приводить бессмысленно, поскольку в отрыве от реальной задачи он не даст исчерпывающего понимания, как делать надо, а как не надо, поскольку то, как надо в одной ситуации, примененное на другую может привести к плачевной ситуации и попа-больному рефакторингу.

### L - Подстанока Лискова (Liskov Substitution)

Принцип, вытекающий из соблюдения принципа Open/Closed - дочерние объекты не должны нарушать логику базового. т.е. дочерний объект должен быть полным корректным аналогом базового класса с точки зрения множества алгоритмов, применяемых к базовому классу. С точки зрения реализации, аналогом он может не являться, расширяя или меняя логику работы базового класса, но он должен оставаться корректным на множестве применяемых алгоритмов.

```c#
// ❌ ПЛОХО: Наследник ломает поведение родителя
класс Bird {
    метод Fly() { ... }
}

класс Penguin : Bird {
    метод Fly() { throw new Exception("Не умею летать") }  // ❌
}

функция MakeBirdFly(Bird b) {
    b.Fly()   // упало для пингвина
}
// ❌ ПЛОХО: спагеттификация, пложение интерфейсов
интерфейс IFlyable {
    метод Fly()
}

интерфейс ISwimmable {
    метод Swim()
}

класс Sparrow : IFlyable {
    метод Fly() { ... }
}

класс Penguin : ISwimmable {
    метод Swim() { ... }
}
// Животные в принципе слабо подходят для аналогий программного кода, поскольку биологическая иерархия не является функциональной иерархией, а в программировании доминантой является функциональная иерархия.
// ✅ ХОРОШО: Иерархия определена функциональными возможностями для оперирования над объектами
класс документ{
    поле название
    поле датаСоздания
    поле источник
    поле назначение
    виртуальный метод ДокументВТекст();
    метод ВернутьМетаданные();
}
класс счёт : документ {
    поле[] позиции
    поле основание
    поле сумма
    поле счётДляОплаты
    переопределённый метод ДокументВТекст();
    переопределённый метод ВернутьМетаданные();
    метод ФормироватьЧек();
}
класс служебнаяЗаписка : документ{
    переопределённый метод ДокументВТекст();
}
класс накладная : документ {
    поле[] позиции
    поле доставленоДатаВремя
    поле приянтоДатаВремя
    поле курьер
    поле ответственный
    переопределённый метод ДокументВТекст();
    переопределённый метод ВернутьМетаданные();
}
/*
Здесь все документы имеют название (для счёта это его номер, для накладной это номер, для служебной записки это заголовок определяющий суть служебной записки), каждый документ имеет дату, каждый имеет источник (кто подал этот документ или создал или отправил), каждый имеет назначение (кому адресовано (компания или фио директора или иного ответственного лица)). 
Каждый можно сериализовать в некоторый текст и из каждого можно получить перечень метаданных. Общие метаданные по принципу Open/Close и Liskov Substitution должны соответствовать базовому классу. (В качестве возвращаемых метаданных могут выступать словари) и алгоритмы, работающие с документами в целом (составление отчётов о документообороте, например) должны работать со всеми документами.
Реализации конкретных документов могут отличаться, но они не должны нарушать принцип ненарушения работы базового класса и быть корректны для алгоритмов на множестве алгоритмов для документов.
Дополнительная функциональность, не определяемая документами в целом, может быть добавлена через реализацию интерфейсов, не затрагивая функциональность основного класса.
*/
```

### I - Interface Segregation (Разделение интерфейсов)

Принцип говорит: интерфейс отвечает за единственное множество функциональности, определённое на функциональном классе. Что это значит? Интерфейс должен предписывать классу реализовать только ту функциональность, которая определяется глаголом в его названии и не должен обязывать реализовать то, что не требуется реализующим его классам.

```c#
// ❌ ПЛОХО: Огромный интерфейс на всё
интерфейс IWorker {
    метод Work()
    метод Eat()
    метод Sleep()
    метод TakeBreak()
}

класс Robot : IWorker {
    метод Work()   { ... }
    метод Eat()    { throw ... }   // не нужен
    метод Sleep()  { throw ... }   // не нужен
    метод TakeBreak() { ... }      // не нужен, но надо реализовать
}
// ✅ ХОРОШО: Разбили на мелкие интерфейсы
интерфейс IWorkable   { метод Work() }
интерфейс IEatable    { метод Eat() }
интерфейс ISleepable  { метод Sleep() }

класс Human : IWorkable, IEatable, ISleepable {
    метод Work()  { ... }
    метод Eat()   { ... }
    метод Sleep() { ... }
}

класс Robot : IWorkable {
    метод Work() { ... }   // только то, что нужно
}
```

### D — Dependency Inversion (Инверсия зависимостей)

Инверсия зависимостей может сбить с толку, при прямолинейной трактовке. Смысл в том, чтобы функциональность, которая имеет множество реализаций зависела не от конкретной реализации, а от интерфейса взаимодействия (абстракции) над этой реализацией. 

```c#
// ❌ ПЛОХО: Высокий уровень зависит от низкого
класс EmailSender {
    метод Send(message) { ... }
}

класс NotificationService {
    private EmailSender sender = new EmailSender()   // жёсткая привязка

    метод Notify(user, text) {
        sender.Send(text)
    }
}
// Чтобы заменить EmailSender на SmsSender — надо менять NotificationService
// ✅ ХОРОШО: Зависим от интерфейса, а не от конкретного класса
интерфейс IMessageSender {
    метод Send(message)
}

класс EmailSender : IMessageSender {
    метод Send(message) { ... }
}

класс SmsSender : IMessageSender {
    метод Send(message) { ... }
}

класс NotificationService {
    private IMessageSender sender

    конструктор NotificationService(IMessageSender s) {
        sender = s                         // DI — внедрение зависимости
    }

    метод Notify(user, text) {
        sender.Send(text)
    }
}
```

Когда нарушаешь: чтобы поменять способ отправки, приходится переписывать NotificationService → тестировать сложно, изменения дорогие

### Интерлюдия

Прочитав принципы программирования SOLID изнутри может начать точить червячок, который намекает упорно, что это бубуйня какая-то и где-то гоев разводят на бабки. 

Таки да, разводят, а они разводиться и согласны. 

Есть 2 стула. 1 стул - пики точёные, на нём сидит бизнес. Бизнесу надо здесь и сейчас, пики это - уходящие клиенты, погоня за рынком, налоговая, забухавший Петрович, застопивший всё производство и класть бизнес хотел на красивый код, чистую ООП архитектуру и дороговизну масштабируемости, если время, потраченное на разработку того, что бизнесу требуется здесь и сейчас не идёт на решение его проблем, а по сути, является решением будущих проблем разработчиков. Чувствуете? Говной воняет. Когда кто-то за твои бабки решает свои будущие проблемы. 2 стул - (сами догадайтесь что), на нём сидит корпорация разработчик ПО, её интерес - экономия, контроль, управление. Архитектура, в которой ни один разработчик своей криворукостью ничего не сломает - хорошая архитектура. Цель такой корпорации не решать проблемы клиентов, а извлекать прибыль, если извлекаемая прибыль частично идёт на то, чтобы в будущем процент маржинальности не упал - для корпорации это благо. 

Ни на одном из этих стульев не сидит разработчик, он сидит и плачет в душе, потому что ему приходится разбираться с сущностями, с которыми он не стал бы разбираться, если бы применил бритву Оккама к программному коду и не наплодил абстракций, которые загоняют его самого в рамки.

Главная проблема в том, что ошибка в выборе архитектуры на ранних этапах может привести к проблемам категории: нужно сделать фичу, а наша архитектура не приспособлена для таких изменений. И наступает анал-карнавал, когда почти всё переписывается с нуля, либо фича деплоится на костыли, но тогда какая чистая архитектура? Какое следование принципам SOLID?

Вот и получается такая неутешительная ситуация, выхода из которой, по сути нет. Есть вариант работать в маленькой аутсорс команде, которая решает задачи бизнеса, обозначая ему, какие риски решения проблемы здесь и сейчас и что даже если мы будем идти по "правильному" пути, то от риска упереться в технические архитектурные ограничения, ПО не застраховано и рано или поздно его придётся переписывать и проводить глубокую переработку. Как с автомобилями, современные двигатели далеко ушли от древних карбюраторных собратьев. Появились инжекторы, датчики, мозги. Ключевой принцип не изменился, но появилась какая-то обвязка, для которой двигатель практически всегда переизобретают.

Ввиду вышеобозначенных причин, существует ряд принципов, которые создали не маркетологи и не корпорации, а народный фольклор программистов, который передаёт эти знания из поколения в поколение, эти принципы сильно проще, чем SOLID и объясняют ситуацию сильно лучше, чем SOLID, и подходят для решения куда большего класса задач, чем SOLID.

### KISS — Keep It Simple, Stupid
Простота — залог успеха. Большинство систем работают лучше, если они простые, а не сложные.

Самый древний инженерный принцип, передававшийся задолго до возникновения программирования и информационных технологий от мастеров к подмастерьям гласит: чем меньше в системе актуаторов, тем меньше вероятность, что что-то сломается. Или проще: меньше телодвижений системы рождают меньше проблем. 

Какой смысл плодить абстракции и усложнять чтение и понимание кода, если можно этого не делать? Можно сделать где-то суперкласс, можно вообще не делать класс, обойтись структурой. Зачем для хранения информации о пользователях класс? Какую роль он решает? Валидация данных? Изменение собственных полей? ну такое. Проще структура и набор методов или класс-оркестратор, который будет управлять пользователями в целом, а не одним конкретным. 

```c#
// Плохо, сложно, непонятно
класс АбстрактныйФабричныйМенеджерПроцессов {
    метод СоздатьЦепочкуОтветственности() {
        // 50 строк кода с паттернами, интерфейсами и делегатами
    }
}

// Хорошо, просто, понятно
функция ОбработатьЗаказ(заказ) {
    проверитьНаличие(заказ)
    списатьТовары(заказ)
    сформироватьЧек(заказ)
}
```

SOLID говорит: «Разделяй на мелкие классы». KISS говорит: «Разделяй, но только когда это реально упрощает». Если для заказа нужны три простые функции — пиши три функции, а не три фабрики, четыре сервиса и пять интерфейсов. Меньше сущностей — проще жизнь.

### DRY — Don't Repeat Yourself
Не повторяйся. Дублирование кода — источник ошибок и усложнения поддержки.

Банальнейшая проблема: где-то есть 5 переписанных строчка в строчку экземпляров одного и того же алгоритма. И тут выясняется, что в нём есть баг или изменились требования - переписывай все 5, а того глядишь и 6 напишешь и так далее. А потом какой-нибудь Петя потеряет один из алгоритмов и вас двоих ждёт увлекательное путешествие в отладку в час ночи, потому что там баг.

```c#
// Плохо. Дублирование, хардкод. 
функция РассчитатьНалогДляФизЛица(сумма) {
    возврат сумма * 0.13
}

функция РассчитатьНалогДляЮрЛица(сумма) {
    возврат сумма * 0.13   // та же логика, но скопирована
}

// Хорошо, без дублирования, но всё ещё хардкод, ставку стоит вынести в отдельный именованный литерал или переменную

функция РассчитатьНалог(сумма, ставка) {
    возврат сумма * ставка
}

// Использование:
РассчитатьНалог(1000, 0.13)  // для физлица
РассчитатьНалог(5000, 0.20)  // для юрлица (если ставка другая)
```

SOLID говорит: «Сделай отдельный класс для каждого налога и интерфейс для расчёта». DRY говорит: «Вынеси общую логику в одну функцию». И знаешь, что проще и понятнее? Функция с параметрами, а не иерархия классов.

Важно: Не переусердствуй. Иногда повторение кода допустимо, если оно улучшает читаемость. Например, три одинаковые строки в разных методах лучше, чем одна переусложнённая абстракция, которую никто не понимает.

### YAGNI — You Aren't Gonna Need It
Не добавляй функциональность, пока она не стала действительно необходимой. Это экономит время и силы.

Принцип из экстремального программирования (XP). Его автор — Рон Джеффрис. Впервые озвучен в конце 1990-х как противоядие от «проектирования на вырост».

**Не делай** того, что не нужно прямо сейчас. Если ты думаешь: «А вдруг нам понадобится поддержка нескольких баз данных?», и добавляешь абстракцию — ты тратишь время, деньги и усложняешь код ради функциональности, которая, скорее всего, никогда не пригодится.

Если реально понадобится, заменить всё на интерфейс по образу и подобию существующего решения будет сильно проще и дешевле в конечном итоге, чем что-то планировать заранее.

```c#
// Проектируем систему для управления задачами
интерфейс IХранилищеЗадач { ... }
класс ХранилищеВSql : IХранилищеЗадач { ... }
класс ХранилищеВMongo : IХранилищеЗадач { ... }  // на всякий случай

// Бизнес: "Нам нужен простой список задач в SQL"

// Хорошо: так делай
класс ХранилищеЗадач {
    // Простая работа с SQL, без интерфейсов
}

// Через полгода бизнес реально попросил MongoDB
// — тогда и сделаешь интерфейс и вторую реализацию
// А как? да просто переименуешь класс и добавишь интерфейс на основе класса, ничего сложного, основная логика не будет затронута
```

SOLID особенно D (Инверсия зависимостей) говорит: «Зависи от интерфейсов, а не от реализации». YAGNI говорит: «Сначала сделай реализацию, а когда она тебе понадобится — тогда и делай интерфейс». Практика показывает, что большинство «гибких» дизайнов никогда не становятся гибкими — они просто усложняют код.

### Separation of Concerns (SoC)
Разделяй ответственность. Каждая часть программы должна решать одну задачу.

Один из старейших принципов инженерии. Ещё в 1970-х Дейкстра и другие пионеры программной инженерии говорили: «Разделяй сложную задачу на маленькие, каждая из которых решает одну чёткую проблему».

Каждая часть программы должна отвечать за одну, и только одну, вещь. Это не «одна причина для изменения» (как в SRP), а естественное разделение на логические слои: интерфейс — отдельно, бизнес-логика — отдельно, работа с данными — отдельно.

И даже в этом делении есть куда делить дальше. Интерфейс это не супер-метод, который в десятке разных мест воздействует на DOM интерфейса, а множество маленьких методов, решающих свои микрозадачи, которые можно описать одним глаголом. Если задачу нельзя описать одним конкретизированным глаголом, значит её надо разбить, абстрактное OperatingData - говорит о том, что внутри должно быть разделение. Да и сам глагол такой себе, никакой конкретики.

```c#
// Плоха, всё в одном
класс Приложение {
    метод Запустить() {
        показатьОкно()               // интерфейс
        загрузитьКонфигИзФайла()     // работа с файлами
        подключитьсяКБазе()          // доступ к данным
        обновитьОтображение()        // опять интерфейс
    }
}
// Хорошо:  программа имеет слои взаимодействия, сломалась база данных - идёшь чинить ТОЛЬКО базу данных, не трогая другой код
класс МенеджерКонфига { ... }
класс БазаДанных { ... }
класс ОтрисовщикИнтерфейса { ... }

класс Приложение {
    конструктор(конфиг, база, отрисовщик) { ... }
    метод Запустить() {
        // каждая часть отвечает за своё
    }
}
```

SRP (S) говорит: «У класса одна причина для изменения». SoC говорит: «Разделяй логические слои». SRP — частный случай SoC. Но SoC шире: он про организацию системы в целом, а не про отдельные классы.

Важно: SoC не означает «дроби всё на атомы». Это значит «группируй по смыслу». Если у тебя три функции, работающие с пользователем — они могут быть в одном классе, потому что это один слой.

### Command Query Separation (CQS)
Метод либо изменяет состояние, либо возвращает данные, но не делает и то, и другое одновременно.

Принцип придумал Бертран Мейер, создатель языка Eiffel, в 1980-х. Он заметил, что смешивание изменения состояния и возврата значения приводит к неожиданным побочным эффектам.

Каждый метод должен быть либо командой (изменяет состояние), либо запросом (возвращает данные), но не одновременно:  
- Команда (void-метод): делает что-то с объектом (устанавливает, удаляет, изменяет).
- Запрос (функция): возвращает данные, но ничего не меняет.

```c#
интерфейс ISteeringWheel {
    метод Повернуть(угол)        // команда
    метод УголПоворота()         // запрос
    метод ПовернутьИВернуть(угол) // ❌ и команда, и запрос
}

интерфейс ISteeringWheel {
    метод Повернуть(угол)        // команда
    метод УголПоворота()         // запрос
}
```

Исключение методы `TryЧто-тоТам`, которые возвращают результат в виде булевого значения, который определяет выполнена операция или нет.

SOLID не говорит про CQS напрямую. Это принцип из другого мира — контрактного программирования. Но CQS идеально ложится в идею чистых функций из функционального программирования. Команды — это мутация состояния, запросы — чистое преобразование данных.

Результат: Код становится предсказуемым. Ты знаешь, что функция ПолучитьДанные() ничего не сломает, а ИзменитьДанные() не вернёт никакого значения, которое ты забудешь обработать.

### Law of Demeter (LoD)
Объект не должен знать о внутреннем устройстве других объектов. Минимизируй связанность.

Принцип сформулировал Ян Холланд в 1987 году в проекте Demeter. Назван в честь богини плодородия (ирония в том, что принцип про ограничения). В народе известен как «Не разговаривай с незнакомцами».

Объект не должен знать о внутреннем устройстве других объектов. Ты не должен писать цепочки вида a.b.c.d() — потому что если b или c изменятся, твой код сломается. Вместо этого объект a должен попросить b сделать что-то, а b сам разберётся, как.

```c#
// Плохо, предполагается внутренняя структура вложенных данных, очень плохо, валидация превращаяется в спагетти, вложенность растёт
заказ.Покупатель.Адрес.Город.ПолучитьИндекс()
// Классно, удобно, просто и понятно. 
заказ.ПолучитьИндексПокупателя()  // внутри сам разберётся
```

SOLID особенно O (Open/Closed) и L (Liskov) поощряют глубокие иерархии. LoD говорит: «Не строй длинные цепочки, они хрупкие». Чем меньше объект знает о других, тем проще его тестировать и менять.

Важно: LoD не запрещает вызовы цепочек, если это необходимо. Он говорит, что это пахнет говной. Если ты видишь a.b.c.d() — скорее всего, у тебя проблема с инкапсуляцией.

### Composition Over Inheritance
Предпочитай композицию наследованию. Это гибче и проще для поддержки.

Принцип стал популярен благодаря книге «Банды четырёх» (Design Patterns, 1994). Они писали: «Предпочитайте композицию объектов наследованию классов».

Лучше собирать поведение из маленьких частей (композиция), чем выстраивать глубокие иерархии наследования. Наследование жёстко привязывает дочерний класс к родительскому, а композиция позволяет гибко менять поведение, подставляя разные объекты.

```c#
// Осуждаем, нарушение в чистом виде. Выделить функциональную иерерахию (а именно она в основе наследования, а не логическая) бывает сложно и вот банальный пример
класс ТранспортноеСредство {
    метод Двигаться()
}

класс Машина : ТранспортноеСредство { ... }
класс Самолёт : ТранспортноеСредство { ... }
класс Квадрокоптер : ТранспортноеСредство { ... }  // а он не ездит, а летает

// Чуть лучше, но не идеально из-за примитивности примера, призванного чисто показать, кавокуда
класс Двигатель {
    метод Запустить()
}

класс Крылья {
    метод СоздатьПодъёмнуюСилу()
}

класс Колёса {
    метод Крутиться()
}

класс Машина {
    двигатель = новый Двигатель()
    колёса = новый Колёса()
    метод Двигаться() { колёса.Крутиться(); двигатель.Запустить() }
}

класс Самолёт {
    двигатель = новый Двигатель()
    крылья = новый Крылья()
    метод Лететь() { двигатель.Запустить(); крылья.СоздатьПодъёмнуюСилу() }
}
```

SOLID особенно O (Open/Closed) и L (Liskov) — это про наследование. Но в реальности наследование часто ломается, особенно когда иерархия становится глубокой. Принцип «Композиция над наследованием» — это признание того, что наследование — это хрупкая связь.

Вывод: Используй наследование только когда это действительно «является» (IS-A): собака — это животное, но для наследования не подходит, ибо IS-A это про поведение, не всё общее функциональное поведение животных можно вывести сразу в логику программы, это надо понимать и для большинства задач композиция даёт больше гибкости, проще "собрать" животное по частям: ротовой аппарат, передние конечности, задние конечности, наличие-отсутствие хвоста и прочее.

## Критика ООП и почему процедурный/функциональный подход часто лучше

### Проблема: ООП как инструмент решения бизнес-задач
Бизнесу нужен работающий продукт здесь и сейчас, а не архитектурный шедевр на вырост. ООП заставляет думать о будущем, а не о текущей задаче, что замедляет разработку.

### Процедурный/функциональный подход: "здесь и сейчас"
- Прямой поток данных: вход → преобразование → выход.
- Минимум косвенности, нет глубоких иерархий.
- Фокус на задаче, а не на структуре.

### Почему ООП не стыкуется с реальностью бизнеса?
Бизнес меняется быстро. ООП с жёсткими иерархиями плохо адаптируется, рефакторинг дорогой и долгий. Процедурный код меняется локально и дёшево.

### Итог
В большинстве проектов процедурный/функциональный подход быстрее, проще и дешевле. ООП имеет смысл только в больших, долгоживущих системах с огромной командой.
