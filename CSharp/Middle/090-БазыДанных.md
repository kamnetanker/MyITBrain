# Основы SQL и работы с базами данных

## Часть 1. Основы управления реляционными базами данных

База данных - организованное хранилище структурированных данных.

Базы данных бывают:

- Реляционные - таблицы со связями зависимости;
- Графовые - узлы со связями (не обязательно зависимости, скорее чаще принадлежности);
- Временных рядов - последовательность время-данные;
- Не-реляционные - JSON хранилища, YAML хранилища;
- Объектные - Бинарные хранилища объектов.

Понятия SQL баз данных:

- База данных - набор связных таблиц
- Таблица - набор строк(записей) и столбцов(полей)
- Строка - запись, состоящая из столбцов (полей)
- Столбец - атрибут, некоторый значимый тип или ссылка на другую таблицу
- Первичный ключ - столбец, содержащий уникальный в рамках таблицы идентификатор, позволяющий недвусмысленно отделить одну запись от другой
- Составной первичный ключ - совокупность из нескольких столбцов, определяющих уникальную запись. По отдельности атрибуты в столбце могут быть не уникальны
- Вторичный ключ(внешний) - Ключ-ссылка на первичный ключ в другой таблице, позволяет организовывать связность таблиц.
- DDL (Data Definition Language) - язык определения структуры. Совокупность команд и синтаксиса для определения структуры таблиц, состав ключей этих таблиц, типы данных столбцов и т.д.: `CREATE`, `ALTER`, `DROP` 
- DML (Data Manipulation Language) - язык манипуляции данными, пресловутое CRUD: `INSERT`, `SELECT`, `UPDATE`, `DELETE`

Основные типы данных:

- Целочисленные (`INT`, `BIGINT`, `SMALLINT`, `TINYINT`) идентификаторы, счётчики
- Строки (`VARCHAR(n)`, `NVARCHAR(n)`, `CHAR(n)`) Имена, тексты
- Дата и время (`DATE`, `DATETIME`, `DATETIME2`) Дата, временные метки
- Денежные (`DECIMAL(p, s)`, `MONEY`) Цены, финансы
- Логические (`BIT`) флаги
- Уникальные (`UNIQUEIDENTIFIER`) GUID
- Бинарные (`VARBINARY`) - файлы, блобы (редко, лучше в другие базы выносить, а в реляционной хранить только "Где взять" эти данные)

### DDL - создание и изменение структуры данных

`CREATE TABLE` - определение новой таблицы в базе данных и подготовка её для приёма дынных

```sql
CREATE TABLE Пользователи (
    Id INT PRIMARY KEY IDENTITY(1,1),   -- автоинкремент[reference:18]
    Имя NVARCHAR(100) NOT NULL,        -- NOT NULL запрещает NULL[reference:19]
    Email VARCHAR(255) UNIQUE,
    ДатаРегистрации DATETIME DEFAULT GETDATE(), -- DEFAULT — значение по умолчанию[reference:20]
    Активен BIT DEFAULT 1
);
```

`ALTER TABLE` - обновление структуры таблицы (добавляет, изменяет, удаляет столбцы и ограничения)

```sql
-- Добавить столбец
ALTER TABLE Пользователи ADD Телефон VARCHAR(20) NULL;

-- Изменить тип
ALTER TABLE Пользователи ALTER COLUMN Имя NVARCHAR(200) NOT NULL;

-- Удалить столбец
ALTER TABLE Пользователи DROP COLUMN Телефон;
```

`DROP TABLE` - удаление таблицы безвозвратно

```sql
DROP TABLE Пользователи; -- Мем с котиком: ТЫ ЧЕВО НАДЕЛАЛ
```

### DML - добавление, изменение, выборка, удаление данных

`SELECT` - извлечение данных из БД (для реляционных БД самая частая операция)

```sql
-- Все столбцы, все строки
SELECT * FROM Пользователи;

-- Конкретные столбцы
SELECT Id, Имя, Email FROM Пользователи;

-- С условием (WHERE)[reference:24]
SELECT * FROM Пользователи WHERE Активен = 1;

-- С сортировкой (ORDER BY)[reference:25]
SELECT * FROM Пользователи ORDER BY ДатаРегистрации DESC;

-- С ограничением (TOP)
SELECT TOP 10 * FROM Пользователи ORDER BY ДатаРегистрации DESC;
```

**Важно**: без `WHERE SELECT` возвращает все строки. Без `ORDER BY` порядок не гарантируется

`INSERT` - вставка данных в таблицу

```sql
-- Одну запись
INSERT INTO Пользователи (Имя, Email, Активен)
VALUES ('Иван Петров', 'ivan@mail.ru', 1);

-- Несколько записей
INSERT INTO Пользователи (Имя, Email)
VALUES 
    ('Мария Смирнова', 'maria@mail.ru'),
    ('Алексей Иванов', 'alex@mail.ru');
```

`UPDATE` - обновление строк в таблице

```sql
-- Конкретную запись
UPDATE Пользователи 
SET Email = 'newemail@mail.ru' 
WHERE Id = 5;

-- Несколько записей
UPDATE Пользователи 
SET Активен = 0 
WHERE ДатаРегистрации < '2023-01-01';
```

**Без `WHERE` обновятся ВСЕ строки!**

`DELETE` - Удаляет данные из таблицы

```sql
-- Конкретную запись
DELETE FROM Пользователи WHERE Id = 10;

-- Все неактивные
DELETE FROM Пользователи WHERE Активен = 0;
```

**Без `WHERE` удалятся ВСЕ строки!**

### NULL - отсутствие значения

`NULL` - означает, что данные неизвестны, но возможно они будут добавлены позже. Это не отстутствие значения и не нулевое, это неизвестность. Никакие два `NULL` не будут равны друг другу, а `NULL` в качестве `ID` недопустимо.

```sql
-- Проверка на NULL (не работает =)
SELECT * FROM Пользователи WHERE Телефон IS NULL;    -- IS NULL[reference:35]
SELECT * FROM Пользователи WHERE Телефон IS NOT NULL; -- IS NOT NULL

-- Замена NULL на значение
SELECT Id, ISNULL(Телефон, 'не указан') AS Телефон FROM Пользователи;
```

## Часть 2. Сложные составные запросы к базе данных

### Фильтрация

`WHERE` - определяет условия, которым должны соответствовать атрибуты записей, чтобы запрос был применён к ним.

```sql
-- Простое равенство
SELECT * FROM Пользователи WHERE Активен = 1;

-- Несколько условий (AND / OR)
SELECT * FROM Пользователи 
WHERE Активен = 1 AND ДатаРегистрации > '2024-01-01';

-- Сравнение с NULL (через IS NULL / IS NOT NULL, НЕ через =)
SELECT * FROM Пользователи WHERE Телефон IS NULL;        -- правильно[reference:3]
SELECT * FROM Пользователи WHERE Телефон = NULL;        -- ошибка, никогда не вернёт строки[reference:4]
```

Количество `AND`, `OR`, `NOT` не ограничено, а так же допустимо использовать `IN`, `BETWEEN`, `LIKE` сколько угодно для более сложной фильтрации данных. 

### Сортировка 

`ORDER BY` - сортировка по указанным атрибутам. 

```sql
-- Сортировка по одному столбцу (по умолчанию ASC — по возрастанию)
SELECT * FROM Пользователи ORDER BY ДатаРегистрации;

-- По убыванию (DESC)
SELECT * FROM Пользователи ORDER BY ДатаРегистрации DESC;

-- Сортировка по нескольким столбцам
SELECT * FROM Пользователи 
ORDER BY Активен DESC, ДатаРегистрации ASC;

-- Можно использовать позицию столбца в SELECT (не рекомендуется для читаемости)
SELECT Id, Имя FROM Пользователи ORDER BY 2;  -- сортировка по Имя (второй столбец)[reference:7]
```

Без `ORDER BY` данные из таблицы придут "как есть".

### JOIN - объединение таблиц и вывода

`JOIN` - позволяет объединить таблицы на основании логических связей между ними.

```sql
CREATE TABLE Пользователи (
    Id INT PRIMARY KEY IDENTITY(1,1),   -- автоинкремент[reference:18]
    Имя NVARCHAR(100) NOT NULL,        -- NOT NULL запрещает NULL[reference:19]
    Email VARCHAR(255) UNIQUE,
    ДатаРегистрации DATETIME DEFAULT GETDATE(), -- DEFAULT — значение по умолчанию[reference:20]
    Активен BIT DEFAULT 1
);

CREATE TABLE Заказы (
    Id INT PRIMARY KEY IDENTITY(1,1),
    ПользовательId INT NOT NULL,
    Сумма DECIMAL(18,2) NOT NULL,
    ДатаЗаказа DATETIME DEFAULT GETDATE()
);

-- INNER JOIN  Возвращает только те строки, где есть совпадения в обеих таблицах

SELECT Пользователи.Имя, Заказы.Сумма, Заказы.ДатаЗаказа
FROM Пользователи
INNER JOIN Заказы ON Пользователи.Id = Заказы.ПользовательId;

-- LEFT JOIN Возвращает все строки из левой таблицы (Пользователи), даже если нет совпадений в правой (Заказы)

SELECT Пользователи.Имя, Заказы.Сумма
FROM Пользователи
LEFT JOIN Заказы ON Пользователи.Id = Заказы.ПользовательId;
-- Пользователи без заказов тоже будут в результате (с NULL в полях Заказы)

-- RIGHT JOIN — все строки из правой таблицы, даже если нет совпадений в левой
-- FULL OUTER JOIN — все строки из обеих таблиц


-- CROSS JOIN Каждая строка из первой таблицы соединяется с каждой строкой из второй. Используется редко, обычно случайно

SELECT * FROM Пользователи CROSS JOIN Заказы;  -- осторожно, может быть очень много строк!
```

### Agregate functions

Агрегатные функции выполняют вычисления над набором значений и возвращают одно (скалярное) значение

- `COUNT(*)` кол-во строк
- `COUNT(столбец)` кол-во не null строк в таблице по столбцу
- `SUM(столбец)` сумма значений 
- `AVG(столбец)` среднее значение
- `MIN(столбец)` минимальное значение
- `MAX(столбец)` максимальное значение

```sql
-- Общее количество пользователей
SELECT COUNT(*) AS ВсегоПользователей FROM Пользователи;

-- Количество пользователей с указанным телефоном (не-NULL)
SELECT COUNT(Телефон) AS СТелефоном FROM Пользователи;

-- Общая сумма всех заказов
SELECT SUM(Сумма) AS ОбщаяСумма FROM Заказы;

-- Средний чек
SELECT AVG(Сумма) AS СреднийЧек FROM Заказы;

-- Минимальный и максимальный заказ
SELECT MIN(Сумма) AS Минимум, MAX(Сумма) AS Максимум FROM Заказы;
```

Важно: агрегатные функции игнорируют `NULL` (кроме `COUNT(*)`)

### Группировка значений

`GROUP BY` группирует результаты по значению одного или нескольких столбцов

```sql
-- Количество заказов по каждому пользователю
SELECT 
    ПользовательId, 
    COUNT(*) AS КоличествоЗаказов,
    SUM(Сумма) AS ОбщаяСумма
FROM Заказы
GROUP BY ПользовательId;

-- С JOIN и GROUP BY
SELECT 
    Пользователи.Имя, 
    COUNT(Заказы.Id) AS КоличествоЗаказов,
    ISNULL(SUM(Заказы.Сумма), 0) AS ОбщаяСумма
FROM Пользователи
LEFT JOIN Заказы ON Пользователи.Id = Заказы.ПользовательId
GROUP BY Пользователи.Имя;
```

Важно: все столбцы в `SELECT`, которые не являются агрегатными, должны быть перечислены в `GROUP BY`

`HAVING` - фильтрация после группировки, работает как `WHERE`, но для групп, а не отдельных строк

```sql
-- Только пользователи с более чем 3 заказами
SELECT 
    ПользовательId, 
    COUNT(*) AS КоличествоЗаказов
FROM Заказы
GROUP BY ПользовательId
HAVING COUNT(*) > 3;
```

## Часть 3. Организация структуры данных

### Нормализация данных

Нормализация - процесс проектирования, используемый для упорядочивания данных в таблицах и столбцах внутри базы данных. Основная цель - минимизировать дублирование данных, что позволяет добиться небольшого прироста производительности в операциях вставки и обновлений. Обычно, применяется 3 формы нормализации, 4-я и 5-я на практике не применяются или применяются очень редко.

`Первая форма нормализации 1НФ`

Требование: 

1. Для каждого набора связанных данных должна существовать отдельная таблица
2. Исключение повторяющихся атрибутов из разных таблиц.
3. Все записи должны быть однозначно идентифицируемы первичным ключом.

`Вторая нормальная форма 2НФ`

Требование одно: если в таблице есть составной ключ, то атрибуты должны зависеть от ключа, а не части ключа.

`Третья нормальная форма 3НФ`

Требование: Все неключевые атрибуты должны нетранзитивно зависеть от первичного ключа. Простыми словами: неключевой столбец не должен зависеть от другого неключевого столбца

Все нормальные формы включают в себя предыдущие, т.е. для выполнения 3 нормальной формы требуется и выполнение 1-й и 2-й

`Человеческим языком`

1. Все атрибуты должны быть связаны и не дублироваться по разным таблицам
2. В таблицах должны быть первичные ключи (составные или индексы)
3. Всё, что не является ключом должно зависеть от ключа, а не от его части(в случае составного), т.е. адреса филиалов зависят от идентификатора филиала, а не от менеджера филиала, следовательно менеджер филиала не входит в ключ.
4. Неключевые столбцы не должны зависеть от других неключевых столбцов. Т.е. менеджер не может зависеть от внешнего ключа по адресам филиалов. Лучше таблица "филиал" с идентификатором, адресом и менеджером через внешний ключ 1к1 + список сотрудников через внешний ключ 1кN.

`Когда нормализация не нужна (денормализация)`

В высоконагруженных системах (отчёты, аналитика) специально нарушают нормализацию для ускорения чтения. Денормализация — это осознанное решение для оптимизации производительности, а не ошибка проектирования.

### Индексы

`Индекс` — это структура на диске или в памяти, связанная с таблицей или представлением, которая ускоряет извлечение строк. Это как оглавление в книге.

Индексы бывают двух основных типов.

`Кластеризованный индекс (Clustered)`

Сортирует и физически хранит строки данных в таблице на основе ключевых значений.

Может быть только один на таблицу, потому что данные можно физически упорядочить только одним способом.

Если у таблицы есть кластеризованный индекс, она называется кластеризованной таблицей.

Если кластеризованного индекса нет, строки хранятся в неупорядоченной структуре, называемой кучей (`heap`).

Обычно создаётся автоматически на `PRIMARY KEY`.

```sql
-- Кластеризованный индекс создаётся вместе с PRIMARY KEY
CREATE TABLE Users (
    UserID INT PRIMARY KEY,  -- автоматически кластеризованный индекс
    Name NVARCHAR(100)
);

-- Явное создание кластеризованного индекса
CREATE CLUSTERED INDEX IX_Users_RegistrationDate ON Users(RegistrationDate);
```

`Некластеризованный индекс (Non-Clustered)`

Имеет структуру, отдельную от строк данных.

Содержит ключевые значения и указатель (`row locator`) на строку с этими значениями.

На одной таблице может быть много некластеризованных индексов.

Указатель (`row locator`) зависит от типа таблицы:

* Для кучи (`heap`) — прямой указатель на строку.
* Для кластеризованной таблицы — ключ кластеризованного индекса.

```sql
-- Создать некластеризованный индекс
CREATE INDEX IX_Users_Email ON Users(Email);

-- Уникальный индекс (запрещает дубли)
CREATE UNIQUE INDEX IX_Users_Email ON Users(Email);
```

**Когда нужны индексы**

Столбец часто используется в `WHERE`, `JOIN`, `ORDER BY`.

Таблица содержит много строк (тысячи+).

Столбец имеет целочисленный тип и уникален или не содержит `NULL`.

**Когда индексы не нужны**

Таблица маленькая (до сотен строк) — сканирование всей таблицы быстрее.

Столбец редко используется в запросах.

Частые `INSERT`/`UPDATE`/`DELETE` — индексы замедляют запись, потому что их нужно обновлять при каждом изменении данных.

Важно: избыточное индексирование или неправильно спроектированные индексы — основная причина проблем с производительностью. Держите индексы узкими (с минимальным количеством столбцов) и не создавайте лишних.

### Диалекты

SQL - конкретный стандарт, а реализации этого стандарта различны и могут привносить свои изменения, улучшения и пр. В данном разделе перечислены некоторые из них и их особенности.

- T-SQL: Microsoft SQL Server, Azure SQL. Отличается от стандарта набором переменных, управляющими структурами, `BEGIN TRANSACTION`, `JSON_QUERY`, `JSON_VALUE`
- PL/SQL: базы данных Oracle. Отличается от стандарта: процурные расширения, блоки, `COMMIT`, `ROLLBACK`, `SAVEPOINT`
- pgSQL (PL/pgSQL): база данных PostgreSQL. По сути полноценный процедурный ЯП, встроенный в PostgreSQL

Несколько примеров.

T-SQL:

```sql
-- T-SQL: переменные и управление потоком
DECLARE @Counter INT = 1;
WHILE @Counter <= 10
BEGIN
    PRINT @Counter;
    SET @Counter = @Counter + 1;
END;
```

PL/SQL:

```sql
-- PL/SQL: блок с объявлением переменной
DECLARE
    v_counter NUMBER := 1;
BEGIN
    LOOP
        DBMS_OUTPUT.PUT_LINE(v_counter);
        v_counter := v_counter + 1;
        EXIT WHEN v_counter > 10;
    END LOOP;
END;
```

pgSQL — PostgreSQL:

```sql
-- pgSQL: блок с объявлением переменной
DO $$
DECLARE
    counter INTEGER := 1;
BEGIN
    LOOP
        RAISE NOTICE '%', counter;
        counter := counter + 1;
        EXIT WHEN counter > 10;
    LOOP;
END $$;
```

### Рекомендации

1. При проектировании базы данных отталкивайся от 3НФ. Далее можешь денормализовать, если это необходимо для производительности
2. Используй минимально необходимые типы данных, не размечай "на вырост" - потребление места вырастет, производительность упадёт, реального профита от заложения сразу больших переменных не будет.
3. Если у тебя 10к запросов по столбцу, ну сделай для него индекс, не ленись. Но не делай много индексов, а то получишь не + к производительности, а тыкву.
4. Избегай выборки всех данных `SELECT *`, старайся всегда указывать требуемые для выполнения запроса столбцы.
5. Старайся использовать ёмкие и короткие названия для столбцов. Если не получается, значит это не 1 столбец.
6. Будь внимателен при UPDATE и DELETE, без WHERE - потеряешь всё и придётся восстанавливать через бекап.
7. Если в столбце мало уникальных значений, но часто используется WHERE - фильтрованные индексы могут поднять производительность.

## Часть 4. C# и базы данных.

### У C# три пути: вебкам, закладки и айти.

1. `ADO.NET` - фундамент всего взаимодействия программ с SQL базами данных. В текущий момент рекомендуется для процедурно формируемых уникальных запросов к базам данных, поддержки legacy кода, для максимальной производительности и контроля. 
2. `Dapper` - средний уровень абстракции над запросами к SQL. Обеспечивает высокую производительность, ручные запросы к базам данных, но есть удобный mapping. Рекомендуется, когда производительность стоит достаточно высоко в приоритетах и не требуется высокая сложность абстракций для кода.
3. `EF Core (Entity Framework)` - полноценная ORM среда, скрывающая от программиста "как это сделано", код пишется быстро, но код сложный, кода мало, но он очень абстрактный. Применяется для маленьких проектов и долгоживущих корпоративных систем. Производительность сильно уступает `Dapper` и `ADO.NET`

Важно: В современном .NET (начиная с .NET Core) асинхронное программирование стало стандартом де-факто. Все три подхода предоставляют как синхронные, так и асинхронные методы.

### ADO.NET - данные есть? а если найду?

Пространство имён `System.Data` и `Microsoft.Data.SqlClient` содержит набор классов системы `ADO.NET` предоставляющих низкоуровневый доступ к SQL запросам к базам данных.

Основные классы:

- `SqlConnection` - соединение с базой данных
- `SqlCommand` - выполнение SQL запроса или хранимой процедуры
- `SqlDataReader` - потоковое чтение результатов

Пример: чтение данных синхронный и асинхронный варианты

```c#
using Microsoft.Data.SqlClient;

string connectionString = "Server=localhost;Database=MyDb;User Id=sa;Password=MyPass;";

// ---------- СИНХРОННЫЙ ВАРИАНТ ----------
using (var connection = new SqlConnection(connectionString))
{
    connection.Open();
    string sql = "SELECT Id, Имя, Email FROM Пользователи WHERE Активен = 1";
    using (var command = new SqlCommand(sql, connection))
    using (var reader = command.ExecuteReader())
    {
        while (reader.Read())
        {
            int id = reader.GetInt32(0);
            string name = reader.GetString(1);
            string email = reader.GetString(2);
            Console.WriteLine($"{id}: {name} ({email})");
        }
    }
}

// ---------- АСИНХРОННЫЙ ВАРИАНТ ----------
using (var connection = new SqlConnection(connectionString))
{
    await connection.OpenAsync();
    string sql = "SELECT Id, Имя, Email FROM Пользователи WHERE Активен = 1";
    using (var command = new SqlCommand(sql, connection))
    using (var reader = await command.ExecuteReaderAsync())
    {
        while (await reader.ReadAsync())
        {
            int id = reader.GetInt32(0);
            string name = reader.GetString(1);
            string email = reader.GetString(2);
            Console.WriteLine($"{id}: {name} ({email})");
        }
    }
}
```

Пример: вставка данных в базу данных.

```c#
// СИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    connection.Open();
    string sql = "INSERT INTO Пользователи (Имя, Email) VALUES (@Имя, @Email)";
    using (var command = new SqlCommand(sql, connection))
    {
        command.Parameters.AddWithValue("@Имя", "Иван Петров");
        command.Parameters.AddWithValue("@Email", "ivan@mail.ru");
        int rowsAffected = command.ExecuteNonQuery();
        Console.WriteLine($"Добавлено строк: {rowsAffected}");
    }
}

// АСИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    await connection.OpenAsync();
    string sql = "INSERT INTO Пользователи (Имя, Email) VALUES (@Имя, @Email)";
    using (var command = new SqlCommand(sql, connection))
    {
        command.Parameters.AddWithValue("@Имя", "Иван Петров");
        command.Parameters.AddWithValue("@Email", "ivan@mail.ru");
        int rowsAffected = await command.ExecuteNonQueryAsync();
        Console.WriteLine($"Добавлено строк: {rowsAffected}");
    }
}
```

Когда использовать ADO.NET:

- Полный контроль над SQL.
- Работа с большими объёмами данных (стриминг через SqlDataReader).
- Legacy-код или высоконагруженные системы, где важна минимальная задержка.
- В асинхронном варианте — для веб-API и UI-приложений, где нужно не блокировать птоки.


### Dapper - я твои данные туда-сюда под забором трусил

`Dapper` — легковесный маппер от команды Stack Overflow. Расширяет `IDbConnection` методами, позволяющими выполнять SQL и автоматически маппить результаты в объекты C#.

Установка: `dotnet add package Dapper`

Первым делом нужна моделька:

```c#
public class Пользователь
{
    public int Id { get; set; }
    public string Имя { get; set; }
    public string Email { get; set; }
    public bool Активен { get; set; }
}
```

Далее уже заполняем модельки:

```c#
using Dapper;
using Microsoft.Data.SqlClient;

string connectionString = "Server=localhost;Database=MyDb;User Id=sa;Password=MyPass;";

// СИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    string sql = "SELECT Id, Имя, Email FROM Пользователи WHERE Активен = 1";
    IEnumerable<Пользователь> users = connection.Query<Пользователь>(sql);
    foreach (var user in users)
        Console.WriteLine($"{user.Id}: {user.Имя} ({user.Email})");
}

// АСИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    string sql = "SELECT Id, Имя, Email FROM Пользователи WHERE Активен = 1";
    IEnumerable<Пользователь> users = await connection.QueryAsync<Пользователь>(sql);
    foreach (var user in users)
        Console.WriteLine($"{user.Id}: {user.Имя} ({user.Email})");
}
```

Пример: вставка данных

```c#
// СИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    string sql = "INSERT INTO Пользователи (Имя, Email) VALUES (@Имя, @Email)";
    var user = new Пользователь { Имя = "Иван Петров", Email = "ivan@mail.ru" };
    int rowsAffected = connection.Execute(sql, user);
}

// АСИНХРОННО
using (var connection = new SqlConnection(connectionString))
{
    string sql = "INSERT INTO Пользователи (Имя, Email) VALUES (@Имя, @Email)";
    var user = new Пользователь { Имя = "Иван Петров", Email = "ivan@mail.ru" };
    int rowsAffected = await connection.ExecuteAsync(sql, user);
}
```

Ну круто же? удобно, производительно. Ляпота.

Нинада писать имена переменных и типов на русском языке, но раз уж примеры в базах данных у нас на русском, то и тут будет на русском.

Есть способы смапить русские столбцы из базы ну и вообще закастомить себе маппинг. Зачем? хз, обычно и там и там модельку делают идентичной, но если всё же надо то вот 5 способов выстрелить себе в колено:

`Псевдонимы` - пихаем прямо в SQL запрос переименования:

```c#
var sql = @"
    SELECT 
        person_id AS Id, 
        first_name AS FirstName, 
        last_name AS LastName 
    FROM Persons";
var users = connection.Query<User>(sql);
```

Это работает всегда, не требует дополнительных настроек, но придётся писать алиасы в каждом запросе

`Сопоставление по соглашению` - назвал я, но вообще это MatchNamesWithUnderscores. Часто в бд исползуется snake_case, потому что админы бд - змеючье логово. Поэтому можно включить автоматическое сопоставление:

```c#
Dapper.DefaultTypeMap.MatchNamesWithUnderscores = true;
```

После этого Dapper будет маппить person_id → PersonId, first_name → FirstName и т.д.

`Type Map` - втупую, напрямую, жестко указать что куда мапить:

```c#
var bookMap = new CustomPropertyTypeMap(
    typeof(Book),
    (type, columnName) =>
    {
        if (columnName == "BookId") return type.GetProperty("Id");
        if (columnName == "BookName") return type.GetProperty("Name");
        throw new InvalidOperationException($"No mapping for {columnName}");
    }
);
SqlMapper.SetTypeMap(typeof(Book), bookMap);
```

Можно пользоваться рефлексией и атрибутами и задать настройку разочек при старте приложения.

`Атрибуты` - через сторонние библиотеки можно расширить функциональность Dapper (Claypool.Dapper.ColumnMapper)

```c#
using Dapper.ColumnMapper;

public class User
{
    [ColumnName("user_id")]
    public Guid Id { get; set; }
    
    [ColumnName("user_name")]
    public string Name { get; set; }
}
```

Один раз при старте атрибуты зарегистрируются и далее работают.

Так же можно использовать атрибуты из `System.Data.Linq.Mapping`:

```c#
[System.Data.Linq.Mapping.Column(Name = "CountryId")]
public int Id { get; set; }
```

`FluentMap` - отдельный класс, который мапит данные из БД, делается так:

```c#
public class ProductMap : EntityMap<Product>
{
    public ProductMap()
    {
        Map(p => p.Name).ToColumn("strName");
        Map(p => p.LastModified).Ignore();
    }
}

// Инициализация при старте
FluentMapper.Initialize(config => 
{
    config.AddMap(new ProductMap());
});
```

`Словарь маппинга`

```c#
var personMapping = new Dictionary<string, Expression<Func<MyPocoClass, object>>>()
{
    { "Id", map => map.Id },
    { "FullName", map => map.Name },
    { "BirthDate", map => map.DateOfBirth }
};
```

Что выбрать?

- Проект маленький, запросов немного - Псевдонимы (AS) — быстро и без лишних зависимостей
- В БД везде snake_case - MatchNamesWithUnderscores = true — одна строка кода
- Нужен сложный маппинг (несколько вариантов) - CustomPropertyTypeMap — полный контроль
- Хочешь атрибуты как в EF - Claypool.Dapper.ColumnMapper или System.Data.Linq.Mapping
- Хочешь держать модель чистой от атрибутов - Dapper.FluentMap
- Нужна типобезопасность - Словарь с выражениями

### EF Core - я твои бд шатал.

Entity Framework Core - полноценная ORM система для .Net. Работа осуществляется строго типизированными объектами, SQL формируется автоматически.

Установка: `dotnet add package Microsoft.EntityFrameworkCore.SqlServer`

Понятия:

- `DbContext` - класс, управляющий подключением и отслеживанием изменений.
- `DbSet<T>` - представляет таблицу результата
- **Entity** - экземпляр класса, соответствующий записям в таблице.

Первым делом необходимо определить модельку данных, по которой будет работать EntityFramework:

```c#
using Microsoft.EntityFrameworkCore;

public class Пользователь
{
    public int Id { get; set; }
    public string Имя { get; set; }
    public string Email { get; set; }
    public DateTime ДатаРегистрации { get; set; }
    public bool Активен { get; set; }
    public ICollection<Заказ> Заказы { get; set; }
}

public class Заказ
{
    public int Id { get; set; }
    public int ПользовательId { get; set; }
    public decimal Сумма { get; set; }
    public DateTime ДатаЗаказа { get; set; }
    public Пользователь Пользователь { get; set; }
}

public class AppDbContext : DbContext
{
    public DbSet<Пользователь> Пользователи { get; set; }
    public DbSet<Заказ> Заказы { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        optionsBuilder.UseSqlServer("Server=localhost;Database=MyDb;User Id=sa;Password=MyPass;");
    }
}
```

Далее уже можно взаимодействовать и что-нибудь таки прочитать:

```c#
// СИНХРОННО
using (var context = new AppDbContext())
{
    var users = context.Пользователи
        .Where(u => u.Активен)
        .OrderBy(u => u.ДатаРегистрации)
        .Select(u => new { u.Id, u.Имя, u.Email })
        .ToList();  // синхронный ToList

    foreach (var u in users)
        Console.WriteLine($"{u.Id}: {u.Имя} ({u.Email})");
}

// АСИНХРОННО
using (var context = new AppDbContext())
{
    var users = await context.Пользователи
        .Where(u => u.Активен)
        .OrderBy(u => u.ДатаРегистрации)
        .Select(u => new { u.Id, u.Имя, u.Email })
        .ToListAsync();  // асинхронный ToListAsync

    foreach (var u in users)
        Console.WriteLine($"{u.Id}: {u.Имя} ({u.Email})");
}
```

А можно и записать, никто как бы не запретит:

```c#
// СИНХРОННО
using (var context = new AppDbContext())
{
    var user = new Пользователь
    {
        Имя = "Иван Петров",
        Email = "ivan@mail.ru",
        ДатаРегистрации = DateTime.Now,
        Активен = true
    };
    context.Пользователи.Add(user);
    context.SaveChanges();  // синхронное сохранение
}

// АСИНХРОННО
using (var context = new AppDbContext())
{
    var user = new Пользователь
    {
        Имя = "Иван Петров",
        Email = "ivan@mail.ru",
        ДатаРегистрации = DateTime.Now,
        Активен = true
    };
    context.Пользователи.Add(user);
    await context.SaveChangesAsync();  // асинхронное сохранение
}
```

А можем и обновить сведения о пользователе:

```c#
// СИНХРОННО
using (var context = new AppDbContext())
{
    var user = context.Пользователи.Find(5);
    if (user != null)
    {
        user.Email = "newemail@mail.ru";
        context.SaveChanges();
    }
}

// АСИНХРОННО
using (var context = new AppDbContext())
{
    var user = await context.Пользователи.FindAsync(5);
    if (user != null)
    {
        user.Email = "newemail@mail.ru";
        await context.SaveChangesAsync();
    }
}
```

Более сложные сценарии, вроде создания миграций, поддержка различных версий и прочее, будет в углубленном конспекте непосредственно по EF Core, здесь базовое использование.

В целом, если ничего сложного или долго поддерживаемого не планируется - EF Core, в противном случае лучше посмотреть в сторону Dapper или ADO.NET.
