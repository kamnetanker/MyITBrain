# Тестирование

`Тестирование` - процесс валидации требований с фактическим поведением системы. Данный процесс является инструментом управления рисками бизнеса, связанными с ошибками в программном обеспечении. Что с обывательского взгляда звучит как "ну упало и упало", для бизнеса может звучать как "оно не работало и мы не заработали N денег". 

Ключевая формула для бизнеса:

> `Риск = Вероятность сбоя х Бизнес последствия`

Где последствия выражаются в денежном эквиваленте, т.е. сколько денег будет потрачено или не будет заработано в результате указанного в формуле сбоя. 

Основа любой стратегии разработки ПО - контроль рисков, в зависимости от размера команды разработки, целей проекта, финансирования, требований к отказоустойчивости, выбираются стратегии тестирования и объём тестирования, поскольку протестировать всё невозможно из-за экономических ограничений.

Тесты можно разделить на 3 подхода:

- `Unit-тесты` - тесты отдельных компонентов системы на соответствие входов и выходов этих компонентов ожидаемым. Условно если у нас есть некоторый сумматор, то набор тестов для него определяется правилами математики и если результат не соответствует ожидаемому, то тестирование провалено.
- `Интеграционные тесты` - тесты системы или нескольких подсистем вместе на выполнение сценариев взаимодействия компонент между собой. Здесь бывает сложно выделить конкретные входы и выходы, поэтому автоматизация такого тестирования не всегда представляется возможной, а результат зависит от собираемых метрик программного продукта.
- `E2E (End-to-end) тесты` - ручное тестирование поведения системы с точки зрения пользователя этой системы. Насколько интуитивно решается задача, насколько просто, насколько система отзывчива, насколько UX соответствует требованиям, насколько он эргономичный и т.д., здесь человек, осуществляющий тестирование, проверяет систему в целом по сценариям взаимодействия пользователя с системой.

Тесты деляться на 3 класса по методологии:

- `Черный ящик` - E2E и интеграционные(в отдельных случаях) тесты с системой в целом, без понимания её структуры.
- `Серый ящик` - интеграционные и E2E(в отдельных случаях), проверка системы с пониманием её структуры, доступом к метрикам, но без взаимодействия с компонентами напрямую.
- `Белый ящик` - Unit-тесты и интеграционные(в отдельных случаях), проверка системы с взаимодействием с её структурой напрямую, дебаггинг и unit-тестирование в частности. Проверяются отдельные компоненты и подсистемы программного продукта.

## Тестирование в проектах с единственным разработчиком

Когда понятие "команда" сводится к понятию "один разработчик" вопрос тестирования стоит наиболее остро, поскольку написание тестов и их проведение требуют времени разработчика и если покрывать тестами всё, как того требует TDD (Test-driven-development), то разработка даже самой простой системы превращается в производственный ад и ПО, вероятнее всего, никогда не увидит свет.

Принцип: тестируется поведение, а не реализация этого поведения, т.е. Unit тесты, за исключением критических компонентов, выбрасываются из планов по разработке, поскольку времени на них уходит много, а результата в виде продвижения в разработке в контексте единственного разработчика, по факту, нет.

---

**Что тестировать?** 

- Правило "золотого пути" - путь, где создаётся потребитель, потребитель решает задачу, потребитель даёт деньги
  - Работа регистрации
  - Работа оплаты или монетизации
  - Работа основной фичи программного продукта
- Единицы тестов E2E на каждый из компонент золотого пути. Есть 2-3 способа регистрации? - значит 2-3 E2E теста и не более. 
- Интеграционные тесты по золотому пути. Проверка регистрации - проверь, что появилась запись в БД, что данные для этого пользователя получаются корректно, что система управления пользователем корректно получает его из БД и отдаёт на фронт.
- Юнит-тесты - только сложные методы, завязанные на деловой логике приложения или взаимодействие с API. 

---

**Что мы не тестируем?** 

- CRUD - бессмысленно, поскольку чаще всего он реализуется шаблонно и переиспользуется от проекта к проекту, он протестирован вдоль и поперёк, тащить снова тесты в новый проект и переписывать их из-за несовпадения количества столбцов между проектами бессмысленно.
- Методы-прокладки если интеграционный тест показывает, что прокладки отработали - не трогай, работают и ладно.
- Всё, что может сдохнуть, но приложение продолжит работать - остаётся без тестирования.

---

**Какой пайплайн в контексте C#?**

1. `dotnet test` перед коммитом (быстрые интеграционные тесты основной логики)
2. CI - в репозитории должен быть настроен запуск тестов после пуша. Приложение должно собраться, упаковаться и встать в целевом окружении.
3. Нет E2E в CI, только проверка staging окружения по золотому пути.

Если у тебя есть время только на один тип тестов — пиши интеграционные. Они дают максимальную уверенность за минимальные затраты 

## Тестирование в малых командах разработчиков

В малых командах (3-10 человек) чаще всего нет выделенного QA специалиста(ов) и разработчики сами пишут тесты. Т.к. это уже не один разработчик и не пара в режиме XP, то возникают проблемы коммуникации, проблемы того, что твой код может сломать чужой и git-workflow, поэтому подход к тестированию меняется.

Принцип: баланс скорости и качества. Команда не может затягивать разработку, поскольку их мало, а время = деньги, но при этом нельзя допускать, чтобы код одних ломал код других, поэтому отлов багов происходит ещё до того, как они попадают в общую ветку проекта, а следовательно одними интеграционными тестами уже не обойтись.

---

**Что тестировать?**

- Покрыть юнит-тестами деловую логику, за исключением примитивных методов (геттеры-сеттеры и что-то крайне незначительное)
- Интеграционные тесты API и основных потоков управления
- Контрактные тесты внешних API - необходимо быть уверенным в том, что вы с поставщиком общаетесь на одном языке
- E2E - всё так же, тестирование по золотому пути.

---

**Сколько тестов?**

- Юнит тесты 50-60% от общего числа тестов. Поскольку в команде на каждую фичу каждый разработчик пишет юнит-тесты, они покрывают бОльшую часть кода и поэтому их много
- Интеграционные 30-40% - здесь мы отлавливаем проблемы взаимодействия компонентов, поэтому их тоже существенное количество
- E2E 5-10% - медленные и дорогие тесты, которые нужны на этапе stagging для проверки, что приложение в целом выполняет то, что от него требуется.

---

**Пайплайн** 

1. Локльные unit-тесты + быстрые интеграционные
2. PR (pull request): все unit тесты + интеграционные. Проходят - мерджим, иначе откат
3. CI/CD на stagging - приложение должно собраться, упаковаться, развренуться в окружении пройти все тесты, включая E2E согласно плану покрытия требований.
4. Продакшн: мониторинг и теневое(smoke) тестирование (проверка, что всё работает)

Тесты, если нет выделенного QA инженера, пишутся разработчиками. Если есть QA, то его задача проводить интеграционные и E2E тесты, а unit-тесты пишут разработчики самостоятельно одновременно с фичами

---

## Тестирование в корпоративной среде

В компаниях, специализирующихся на IT есть отдел разработчиков, отдел тестировщиков (QA), отдел девопсов, системных администраторов и т.п. и в этой связи появляются руки и головы для реализации сложных стратегий тестирования, которые позволяют создавать высококачественный протестированный продукт, надёжность которого подтверждается тестами. (Другой вопрос в том, что заказчик продуктов у такой компании может желать сэкономить на разработке и тесты будут сделаны на уровне единичного разработчика, но это уже нюансы реальности)

Принцип: проблема тестов - архитектурная проблема, методологическая проблема, проблема надёжности и исполнения обязательств перед заказчиком. Требуется системный подход и тут рождается TDD (Test-driven-development), поскольку ситуация "прод лёг, потому что Вася дропнул базу случайно в коде" - недопустима, поскольку Вася никогда не расплатится с убытками от такого "дропа базы".

---

**Что тестируется?**

- Всё, что обладает деловой ценностью, но выстраиваются приоритеты, где шаблонные методы встают в конце очереди на тестирование.
- Регрессионные тесты - много регрессионного тестирования, постоянное регрессионное тестирование, но полностью автоматическое
- Нагрузочное тестирование, если есть требования к производительности
- Безопаность (OAuth, JWT, авторизация, утечки данных)
- Контрактное тестирование (API между всеми сервисами)

---

**Распределение тестов**

- Юнит тесты 70% - буквально каждая фича покрывается тестами
- Интеграционные 20%
- E2E 10%

---

**Пайплайн**

- Локльные юнит-тесты
- PR тесты - полный прогон всех тестов
- CI тесты - полный прогон старых и новых тестов
- Stagging E2E тесты + ручное, если позволяет бюджет
- Прод: smoke-тесты + мониторинг

---

**Кто пишет тесты?**

- каждый разработчик под свои методы пишет unit
- отдел QA пишет тест-план и параллельно разработке самого продукта реализует систему, которая покроет его тестами на 100%, прежде чем фича будет реализована, на неё готовится тест-план и пишутся автоматические тесты
- QA инженеры проводят аудит системы в E2E

---

## Логика написания тестов

**Шаг 1** Анализ и планирование тестирования

- Изучаются требования заказчика, пайплайн взаимодействия пользователя в системе, техническая документация среды
- Определяется, что тестируется конкретно, какие модули, какие системы, какие входы и выходы
- Выявление рисков. Если сломаются уведомления о том, что какой-то другой отдел проводит совещание добровольное для укрепления командного духа - около нулевые потери, а если сломается отправка документов поставщику - вот тут уже полетят головы.
- Выстраивание приоритетов тестирования - какие сценарии тестируются в первую очередь(золотой путь), а на какие можно подзабить (аватарка в корпоративной системе)
- Определение подхода к каждому сценарию тестирования: ручное (серый ящик), автоматическое(белый ящик) или исследовательское (черный ящик)
- Создание тест-кейсов (чек-листы того, что и как должно работать) для тестов

**Шаг 2** написание тестов (параллельно с разработкой или до неё)

- Написание автоматических тестов
- Ручные тест-кейсы для сложных сценарией
- Настройка тестовых данных и окружения

**Шаг 3** Выполнение тестов

- Запускает тесты локально (во время разработки).
- Интегрирует тесты в CI/CD пайплайн (автоматический запуск).
- Выполняет ручное тестирование (исследовательское, exploratory) для сценариев, которые сложно автоматизировать

**Шаг 4** анализ результатов

- Сверка сколько успешно, сколько провально
- Проверка провальных тестов на соответствие (баг или тест не корректен?)
- Для всего, что баг формируются issue - точное описание среды, последовательности шагов для воспроизведения, логи об ошибках
- Потребность сверить требования при изменении с текущими тестами и обновление для соответствия требованиям

**Шаг 5** Рефакторинг и поддержка тестов

- Удаление устаревших тестов
- Оптимизация тестов
- Обновление тестов под требования

## Инструменты тестировщика

Инструменты:

- Для каждого ЯП существуют библиотеки юнит тестов, для C# это `xUnit`/`NUnit`/`MSTest`
- Иногда в коде требуется сделать заглушки - мок объекты (mocks) здесь используются `Moq`/`NSubstitute`
- Тестовые контейнеры (`Test Containers`) - в `CI/CD` пайплайне собирается тестовый контейнер для интеграционных тестов
- `Playwright`/`Selenium` - автоматизированный `UI`, для автоматических `E2E` тестов
- `CI/CD` (`GitHub Actions`, `GitLab CI`, `Azure DevOps`) - автоматические пайплайны сборки, доставки, тестирования и развёртывания
- Баг-треккеры (`Jira`, `YouTrack`, `GitHub Issues`)

Практики:

- `TDD` (`Test-driven-development`) - тест пишется до того, как будет написан код, который должен пройти тест. Если есть отдельный отдел, который поставляет тесты, как чёрный ящик для разработчика и разработчик не может посмотреть, что за тесты выполняются (чтобы обмануть их), то такая дисциплина разработки имеет право на существование, а если ты, как в анекдоте "сам стучишь, сам отвечаешь, сам себя проверяешь", то это дичь и так делать не надо. Пока в проекте, над которым осуществляется работа нет отдельного QA - TDD это оверхед по ресурсам разработчиков. 
- `Shift-Left` - тестирование начинается ещё на этапе планирования и проектирования. 
- `Risk-Based Testing` - тестирование по приоритетам рисков
- `Continuous Testing` - тесты должны запускаться автоматически и встроены в `CI/CD` пайплайн
- `Contract Testing` - сервисы должны соблюдать контракты взаимодействия

## xUnit - юнит-тесты для C#

xUnit - идейный потомок NUnit и MSTest, был разработан с нуля с учётом опыта последних. Разработчики: Брэд Уилсон и Джеймс Ньюкирк. 

Поддержка: .Net 8 и выше и .Net Framework 4.7.2 и выше, существует интеграция для vs code, visual studio, JetBrains Rider, NCrunch, CodeRush и др.

Данный продукт не является официальным продуктом Microsoft, однако признан отраслевым стандартом и входит в состав .Net Foundation

## Установка

Первым делом надо поставить шаблоны для проектов

```bash
dotnet new install xunit.v3.templates
```

Далее создаётся тестовое пространство

```bash
mkdir MyUnitTests
cd MyUnitTests
dotnet new xunit3
```

Создаётся шаблонный проект со ссылкой на `xunit.runner.visualstudio` (для работы `dotnet test` и интеграции с Visual Studio), простой пустой файл `xunit.runner.json` для настройки раннера тестов, один пример теста.

Далее созданный проект (не решение, проект) должен получить ссылку на то, что тестируется

```bash
dotnet add reference ../MyApp.Core/MyApp.Core.csproj
```

или можно вручную вписать в `.csproj` 

```xml
<ItemGroup>
  <ProjectReference Include="..\MyApp.Core\MyApp.Core.csproj" />
</ItemGroup>
```

**А как писать тесты?**

Необходимо задать `атрибуты`, задать `assert`, `skip`'нуть лишнее и запустить `dotnet test` далее по порядку.

## Атрибуты

`[Fact]` - базовый атрибут для тестов без параметров. Один метод - один тестовый кейс.

```c#
[Fact]
public void Add_TwoNumbers_ReturnsSum()
{
    var calculator = new Calculator();
    var result = calculator.Add(2, 3);
    Assert.Equal(5, result);
}
```

`Theory` - атрибут для параметризованных тестов, один тестовый метод реализует множество тест-кейсов, работающих по одному алгоритму, но с разными данными.

```c#
[Theory]
[InlineData(1, 2, 3)]
[InlineData(-1, 1, 0)]
[InlineData(0, 0, 0)]
public void Add_WithVariousInputs_ReturnsExpected(int a, int b, int expected)
{
    var calculator = new Calculator();
    var result = calculator.Add(a, b);
    Assert.Equal(expected, result);
}
```

Если данные статичны и не меняются - `[Fact]`, если же существует множество тестовых данных - `[Theory]`

## Параметры тестов `[InlineData]`, `[MemberData]`, `[ClassData]`

`[InlineData]` - данные прямо в атрибуте

```c#
[Theory]
[InlineData(2, 3, 5)]
[InlineData(-5, 5, 0)]
[InlineData(100, 200, 300)]
public void Add_MultipleCases_ReturnsCorrectSum(int a, int b, int expected)
{
    var result = new Calculator().Add(a, b);
    Assert.Equal(expected, result);
}
```

Создаётся 3 тест-кейса с набором данных, которые подставляются в параметры тестового метода при прогоне. 

`[MemberData]` - данные из статического свойства или методы

```c#
public static IEnumerable<object[]> AdditionData()
{
    yield return new object[] { 1, 2, 3 };
    yield return new object[] { -1, 1, 0 };
    yield return new object[] { 0, 0, 0 };
}

[Theory]
[MemberData(nameof(AdditionData))]
public void Add_FromMemberData_ReturnsExpected(int a, int b, int expected)
{
    var result = new Calculator().Add(a, b);
    Assert.Equal(expected, result);
}
```

По сути подходит любая структура данных, реализующая `IEnumerable<object[]>`, но пример с методом через `yield` показан, поскольку тестовые данные можно формировать в методе любым доступным способом - можно грузить из `API`, можно из `базы данных`, можно генерировать случайные.

`[ClassData]` - Данные из отдельного класса

```c#
public class AdditionTestData : IEnumerable<object[]>
{
    public IEnumerator<object[]> GetEnumerator()
    {
        yield return new object[] { 1, 2, 3 };
        yield return new object[] { -1, 1, 0 };
    }

    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}

[Theory]
[ClassData(typeof(AdditionTestData))]
public void Add_FromClassData_ReturnsExpected(int a, int b, int expected)
{
    var result = new Calculator().Add(a, b);
    Assert.Equal(expected, result);
}
```

Подходит, когда одного метода не хватает для формирования данных из-за сложности их получения. 

## Assert - сравнение выходов

`Assert` - статический класс, реализующий методы проверки условий. Если условие выполнено - тест пройден, иначе провален. Многие из методов ниже:

- `Assert.Equal(expected, actual)` - Равенство ожидаемого и реального
- `Assert.NotEqual(expected, actual)` - Не равенство ожидаемого и реального
- `Assert.True(condition)` - Условие истинно
- `Assert.False(condition)` - Условие ложно
- `Assert.Null(object)` - Объект равен `null`
- `Assert.NotNull(object)` - Объект не `null`
- `Assert.Throws<T>(()=>action)` - Проверка, что на некорректные данные метод возвращает исключение
- `Assert.Empty(collection)` - Коллекция пуста
- `Assert.NotEmpty(collection)` - Коллекция не пуста
- `Assert.Contains(item, collection)` - Коллекция содержит элемент
- `Assert.DoesNotContain(item, collection)` - Не содержит элемент

Если примеры со сравнением просты для понимания, то с исключениями рассмотрим отдельно

```c#
[Fact]
public void Divide_ByZero_ThrowsDivideByZeroException()
{
    var calculator = new Calculator();
    Assert.Throws<DivideByZeroException>(() => calculator.Divide(10, 0));
}
```

и с коллекцией

```c#
[Fact]
public void GetUsers_ReturnsNonEmptyList()
{
    var service = new UserService();
    var users = service.GetUsers();
    Assert.NotEmpty(users);
    Assert.Contains(users, u => u.Id == 1);
}
```

## Жизненный цикл тестов

`xUnit` создаёт для каждого теста новый экземпляр класса для тестов, что гарантирует изоляцию тестов

```c#
public class CalculatorTests
{
    private readonly Calculator _calculator;

    public CalculatorTests() // Выполняется перед КАЖДЫМ тестом
    {
        _calculator = new Calculator();
    }

    [Fact]
    public void Add_ReturnsSum()
    {
        var result = _calculator.Add(2, 3);
        Assert.Equal(5, result);
    }
}
```

`[IDisposable]` - очистка после каждого теста

```c#
public class DatabaseTests : IDisposable
{
    private readonly Database _db;

    public DatabaseTests()
    {
        _db = new Database();
        _db.Connect();
    }

    public void Dispose() // Выполняется после КАЖДОГО теста
    {
        _db.Disconnect();
        _db.Dispose();
    }

    [Fact]
    public void Query_ReturnsData()
    {
        var result = _db.Query("SELECT * FROM Users");
        Assert.NotEmpty(result);
    }
}
```

`IClassFixture<T>` - если требуется использовать общий контекст для нескольких тестов

```c#
// Фикстура — тяжёлый объект, который создаётся один раз
public class DatabaseFixture : IDisposable
{
    public Database Db { get; }

    public DatabaseFixture()
    {
        Db = new Database();
        Db.Connect();
    }

    public void Dispose()
    {
        Db.Disconnect();
        Db.Dispose();
    }
}

// Тестовый класс использует фикстуру
public class DatabaseTests : IClassFixture<DatabaseFixture>
{
    private readonly Database _db;

    public DatabaseTests(DatabaseFixture fixture) // xUnit внедряет фикстуру
    {
        _db = fixture.Db;
    }

    [Fact]
    public void Query_ReturnsData()
    {
        var result = _db.Query("SELECT * FROM Users");
        Assert.NotEmpty(result);
    }
}
```

## Иногда тесты надо пропускать

Тест, который проверяет ещё не реализованные фичи или заведомо известно, что фича не работает и надо проверить другие фичи, может быть пропущен с помощью атрибутного параметра `Skip = "Reason"`

```c#
[Fact(Skip = "Баг ещё не исправлен, вернуться позже")]
public void BuggyTest_ShouldBeFixed()
{
    // ...
}

[Theory(Skip = "Нестабильные данные")]
[InlineData(1)]
[InlineData(2)]
public void UnstableTest(int value)
{
    // ...
}
```

Сторонние библиотеки могут добавить поддержку `SkippableFact`, которая позволит пропускать по условию, но обычно это не требуется.

Если тест уже длительное время скипается, или он тестировал удалённую функциональность или тест на переработке, но до сих пор висит в `Skip` - удали тест, хватит его скипать, он не нужен.

## Mock - вместо тысячи вызовов

`Mock-объекты` - заглушки, поведение которых настраивается, чтобы имитировать взаимодействие тестируемого класса с другими классами, поскольку классы в реальных приложениях редко являются изолированными. В таких случаях создаются Mock объекты, для которых программным путём задаётся поведение. Условно говоря, мы определяем с какими параметрами вызывается метод этой заглушки и что он вернёт на этот вызов, хардкодом, реальной логики в этом нет, просто хардкод вызов-ответ. + валидируем, что ожидаемые вызовы произошли. 

`Mock` объекты не идут в поставке с xUnit v3, их требуется ставить отдельно библиотекой Moq

`dotnet add package Moq --version 4.20.72` 

Версия может отличатся, но сейчас 4.20 поддерживает .NET 8 и .NET 9, Moq 5 пока в разработке. 

Далее как пользоваться этими Mock объектами.

```c#
using Moq;

// Создаём мок интерфейса IUserService
var mockService = new Mock<IUserService>();

// Или мок абстрактного класса
var mockLogger = new Mock<ILogger<UserController>>();

var service = mockService.Object;

// Настройка возвращаемого значения (синхронно)
// Настраиваем метод GetUser на возврат конкретного пользователя
mockService.Setup(s => s.GetUser(5)) // Пишем вызов
           .Returns(new User { Id = 5, Name = "Иван" }); // а тут хардкодим, что вернёт

// Настройка возвращаемого значения (асинхронно) — важно!
// ✅ Правильно (асинхронно, безопасно)
_mockRepo.Setup(r => r.GetByIdAsync(It.IsAny<int>()))
         .ReturnsAsync(new User { Id = 1, Name = "Иван" });

// ❌ Неправильно (может привести к deadlock)
_mockRepo.Setup(r => r.GetByIdAsync(It.IsAny<int>()))
         .Returns(Task.FromResult(new User { Id = 1, Name = "Иван" }));

// Для методов, возвращающих ValueTask<T>: Moq также поддерживает ReturnsAsync и ThrowsAsync для ValueTask<TResult>

// Настройка исключения
// Синхронное исключение
mockService.Setup(s => s.GetUser(999))
           .Throws(new UserNotFoundException("Пользователь не найден"));

// Асинхронное исключение (для Task / ValueTask)
mockService.Setup(s => s.GetUserAsync(999))
           .ThrowsAsync(new UserNotFoundException("Пользователь не найден"));

// Настройка с использованием It.IsAny<T>
// Когда аргумент не важен или может быть любым:
mockService.Setup(s => s.GetUserAsync(It.IsAny<int>()))
           .ReturnsAsync(new User { Id = 1, Name = "Любой" });

// Теперь при вызове с любым ID вернётся один и тот же пользователь

// Настройка с использованием It.Is<T>(predicate)
// Когда нужно проверить аргумент по условию
mockService.Setup(s => s.GetUserAsync(It.Is<int>(id => id > 0))) // Если id положительный - всё ок, иначе эксепшн
           .ReturnsAsync(new User { Id = 1, Name = "Положительный" });

// Проверка вызовов: Verify, VerifyAll, Verifiable

// Verify — проверка конкретного вызова
// Проверяем, что метод был вызван с правильным аргументом ровно 1 раз
mockService.Verify(s => s.GetUserAsync(5), Times.Once);

// Проверяем, что метод НЕ был вызван
mockService.Verify(s => s.GetUserAsync(It.IsAny<int>()), Times.Never);

// Verifiable + VerifyAll — проверка всех ожидаемых вызовов
// Иногда нужно не просто проверить один конкретный метод, а убедиться, что все ожидаемые вызовы действительно произошли. Для этого используется комбинация Verifiable() и VerifyAll()
// Настройка с пометкой Verifiable
_mockEmail.Setup(e => e.SendAsync(It.IsAny<string>(), It.IsAny<string>(), It.IsAny<string>()))
          .Returns(Task.CompletedTask)
          .Verifiable(); // помечаем как обязательный вызов

_mockRepo.Setup(r => r.GetByIdAsync(It.IsAny<int>()))
         .ReturnsAsync(new User { Id = 1, Name = "Иван" })
         .Verifiable(); // тоже помечаем

// В тесте
await _userService.GetUserAndSendEmailAsync(1, "test@mail.ru");

// Проверяем, что все Verifiable-вызовы были выполнены
_mockEmail.VerifyAll();
_mockRepo.VerifyAll();

// Если какой-либо из помеченных Verifiable() методов не был вызван, VerifyAll() выбросит исключение, и тест упадёт
```

Все `setup` и `verify` происходят непосредственно в методе самого теста, а вот настройка `mock-объекта` на то, поведение какого класса он будет симулировать происходит в конструкторе тестового класса.

```c#
public ClassFullOfTests(){
  _myMock = new Mock<MockableClass>();
}
[Fact]
public async Task TestMyMethod(){
  _myMock.Setup(r = r.GetSomethingByIdAsync(1))
                      .ReturnsAsync(new SomeData{Id = 1, Name= "smthing"});
                      // Тут тестируем
  _myMock.Verify(r = r.GetSomethingByIdAsync(1), Times.Once); // Проверяем что вызвано 1 раз, иначе тест провален
}
```

## Когда тестов слишком много (и как это понять)

Для соло-разработчика правило простое:

> **Если кода тестов больше, чем кода продукта — ты переусердствовал.**

Исключение — сложная бизнес-логика, критические системы или командная разработка. В остальных случаях интеграционные тесты по золотому пути дают максимальную уверенность за минимальные затраты.
