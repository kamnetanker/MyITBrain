# Исключения  

Исключения или исключительные ситуации - ошибки, возникающие в ходе работы программы, которые отправляют обратно по стеку вызовов объект-исключение, который содержит информацию о случившейся ошибке. К примерам таких ошибок можно отнести исключение "Деление на ноль", которое можно перехватить соответствующим обработчиком и вернуть работу программы в корректное русло или сообщить разработчику о том, что в программе необходимо исправить баг. Далеко не все баги можно отловить исключениями, однако данный механизм очень помогает в отладке программ.

- `try-catch-finally` и `throw` - ключевые слова, которые позволяют работать с исключениями

    ```csharp
    try { // Участок кода, где может возникнуть исключительная ситуация
      // code
      throw new Exception("Some Exception"); // отправка исключения
    }catch(Exception ex){// Блок обработки пойманного исключения
      // Тут чота делаем, чтобы было хорошо
    } finally {
      // Блок, срабатывающий всегда, даже если блок try был покинуть командами break, return, goto, он обрабатывается до возврата значения. Необходим, чтобы освободить неуправляемые ресурсы (управление графикой, подключение к БД, файловые потоки и т.п.) без ожидания очередного прохода сборщика мусора.
    }

    ```

    Блоки `catch` при возникновении исключительной ситуации не обрабатываются все, обрабатывается первый подходящий блок, таких блоков может быть множество и каждый может отлавливать не универсальный `System.Exception`, являющийся родительским классом для всех исключений, а какой-либо определённый тип исключений, например `DirectoryNotFoundException` - он наследуется от `System.Exception`, и для него можно создать собственный обработчик `catch (DirectoryNotFoundException ex)` и если после него расположить блок универсальный `catch (Exception ex)`, то универсальный блок уже не будет обрабатывать эту ошибку, а после всех обработок обработается блок `finally`, если он есть.

    Согласно рекомендациям Microsoft: Если при возникновении исключительной ситуации невозможно восстановить работу кода, то не следует перехватывать и обрабатывать это исключение, следует отправить его далее по стеку вызовов (т.е. просто не перехватывать) и код выше по стеку будет определять, можно ли восстановить корректную работу после возникновения данного исключения.

    Пример: если в ходе работы с вводом-выводом в функции возникла ошибка потока, которую не ожидал программист и которой не случалось ранее (например, внезапно умерло блочное устройство ввода-вывода), то код, читающий ввод вывод и имеющий блок `try-catch-finally` не должен обрабатывать данную ошибку, он должен отправить её далее, чтобы код, выделявший ресурсы для совершения операции ввода-вывода принимал решение о том, что делать при возникновении неожиданной исключительной ситуации, например попробовал заново открыть поток и передать его в функцию, осуществляющую ввод-вывод или завершился с сообщением в стандартный поток ошибок системы о том, что блочное устройство недоступно.

- Создание исключений.  

    Объекты-исключения создаются с помощью ключевого слова `throw` внутри любого блока кода.

    ```csharp
    if (shapeAmount <= 0)
    {
        throw new ArgumentOutOfRangeException(nameof(shapeAmount), "Shape amount must be positive.");
    }
    ```

    Выражение в `throw` должно быть неявно преобразуемо в `System.Exception`, т.е. возвращать А) объект Б) объект класса, наследующего Exception

    В стандартной библиотеке исключений есть 2 очень удобных исключения, добавленных туда разработчиками:

  - `ArgumentNullException.ThrowIfNull(param)` — проверяет параметр на null и выбрасывает исключение
  - `ArgumentException.ThrowIfNullOrEmpty(param)` — проверяет строку на null или пустоту

    Если внутри блока `catch` использовать `throw;` без указания объекта, то исключение отправится далее по цепочке с сохранением стека вызовов (т.е. места возникновения) если же заново отправить даже тот же `Exception`, что уже был перехвачен, то в `StackTrace` будет указано место возникновения `catch-блок`.

    ```csharp
    try { ... }
    catch (Exception ex)
    {
        Log(ex);
        throw;   // правильно — стек сохраняется
        // throw ex;   // неправильно — стек теряется
    }
    ```

- Пользовательские исключения  

    Помимо исключений, добавленных разработчиками платформы, можно добавить собственные исключения, унаследовав класс `System.Exception` для создания собственных исключений.

    ```csharp
    public class CustomException : System.Exception
    {
      // конструктор класса по умолчанию
      public CustomException() {}

      // Конструктор с сообщением
      public CustomException(string message) : base (message){}

      // С сообщением и внутренним исключением
      public CustomException(string message, Exception inner) : base(message, inner){} 
    }
    ```

    Создавать и выбрасывать исключения следует, когда:

  - Метод получил недопустимые значения параметров
  - Некорректные вызовы, на основании состояния объекта (запись в read-only, например)
  - Получено исключение из нижележащего слоя, которое оборачивается в более специфичное и передаётся исходное, как inner.

- Фильтры исключительных ситуаций с использованием `when`

    Начиная с C# 6 можно фильтровать исключения используя условные конструкции с ключевым словом `when`

    ```csharp
      try {
        // ...
      }
      catch (SocketException ex) when (ex.SocketErrorCode == SocketError.ConnectionReset){
        // Обработка сброса подключения
      }
      catch (SocketException ex) when (ex.SocketErrorCode == SocketError.TimedOut){
        // Обработка таймаута
      }
    ```

- Производительность

    Исключения - затратная по ресурсам системы операция и следует избегать обработки исключений там, где это А) возможно Б) часто происходят исключения

    Избежать исключительных ситуаций, чаще всего, можно с использованием условных конструкций.

    Для типов, которые реализуют `IDisposable` рекомендуется использовать `using`, вместо блока `finally`. `finally`, в отличие от `using` вызывает очистку всех ресурсов, а не только реализующих `IDisposable`.

    Так же начиная с версии C# 8 можно использовать `using declaration`, которая освободит ресурсы, когда область видимости покинет переменную. Область видимости определяется блоком кода, обёрнутым в `{}` - фигурные скобки. Это может быть условие, цикл, функция, класс, пространство имён. Т.е. если переменная с `using` объявлена внутри условия и выполнение выходит за пределы блока `if`, то, что объявлено внутри с `using` пройдёт процедуру освобождения ресурсов через `IDisposable`

    ```csharp
    using StreamReader reader = File.OpenText("numbers.txt");
    string content = reader.ReadToEnd();
    ```

    В вышеприведённом примере, как только произойдёт выход из области видимости, где объявлен `reader` для него будет вызван метод `Dispose()`, поскольку StreamReader реализует `IDisposable`.

- Рекомендации по использованию
  
  - Используй отлов исключений только тогда и только там, где можешь восстановить работоспособность программы
  - Лови исключения, начиная с более специфичных и заканчивай общими
  - Освобождай ресурсы через `using` или `finally`
  - Если создал исключение - опиши его кратко, но исчерпывающе. Общее описание "что-то пошло не так" не помогает в отладке проблем.
  - Не перехватывай `System.Exception`, если не можешь обработать и вернуть нормальное исполнение программы
  - Не подавляй ошибки пустыми `catch` - усложняет отладку, снижает производительность.
  - Не используй исключения для управления потоком выполнения. Вместо этого используй специфичные Try-функции. (напр. TryGetValue для Dictionary, если не уверен, что ключ есть в коллекции)
  - Не вызывай общий `System.Exception` без основания.
  - Не логгируй и не подавляй исключения без веской причины. Если логгируешь, но не исправляешь - отправь `Exception` дальше используя `throw;`
