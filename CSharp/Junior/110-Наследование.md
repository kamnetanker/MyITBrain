# Наследование

Наследование - механизм ООП позволяющий создавать новые классы на основе уже существующих. Новый класс является производным или "наследником", он наследует все члены родительского (базового) класса и может добавлять свои или изменять базовые, переопределяя их поведение.

C# поддерживает одиночное наследование (только 1 базовый класс)

Неявно все классы наследуют `System.Object`

- Базовый синтаксис наследования

    ```csharp
    class Base
    {
      public int BaseVariable;
    }
    class Derived:Base
    {
      public int DerivedVariable;
    }

    Derived variable = new Derived();
    variable.BaseVariable = 7;
    variable.DerivedVariable = 9;
    Console.WriteLine($"Переменная, унаследованная из базового класса: {variable.BaseVariable} и собственная переменная класса: {variable.DerivedVariable}");
    ```

- Модификаторы доступа при наследовании:
  - `public` - доступен везде
  - `protected` - доступен внутри класса или наследниках
  - `internal` - доступен внутри сборки (assembly)
  - `protected internal` - доступен внутри сборки и только в самом классе и наследниках
  - `private protected` - внутри класса и наследниках класса в рамках сборки
  - `private` - только в классе, но не в наследниках

- `virtual` - ключевое слово, помечающее методы, события или индексаторы как переопределяемые в производных классах. Такие члены имеют реализацию в базовом классе, но наследник может переопределить её.
  - `virtual` нельзя использовать с `static`, `private`, `sealed`
  - Виртуальные члены класса вызываются в соответствии с реальным, а не приведённым типом данных, если используется ключевое слово `override` в классе-наследнике, иначе используется динамическое определение на основании приведённого типа.

    ```c#
    class Base
    {
      public virtual void SayMyName() => Console.WriteLine("Walter White");
      public virtual void AnotherConsoleLog() => Console.WriteLine("Boop");
    }
    class Derived:Base
    {
      public override void SayMyName() => Console.WriteLine("You god damn right!");
      public void AnotherConsoleLog() => Console.WriteLine("weebidy boop");
    }

    
    Base variable = new Derived(); 
    Console.WriteLine($"Say My Name: ");
    variable.SayMyName(); // You god damn right!
    variable.AnotherConsoleLog(); // Boop
    ((Derived)variable).AnotherConsoleLog(); // weebidy boop
    ```

- `abstract` - класс или член класса, не имеющий реализации в базовом классе и обязан быть реализован наследниками. Праобраз контрактов из интерфейсов.
  - Для абстрактных классов
    - Нельзя создать экземпляр (`new`), но созданных наследников можно привести к типу базового абстрактного класса
    - Могут содержать абстрактные и не абстрактные члены
    - Служат шаблонами для классов, имеющих общий контекст, но различную реализацию.
  - Для абстрактных методов
    - Не имеют тела метода, только сигнатура, оканчивающаяся символом `;` вместо блока `{}`
    - Могут быть только в абстрактных классах
    - Неявно являются `virtual`
    - обязаны быть `override` в классах-наследниках
  
  ```c#
  abstract class Payment
  {
      public abstract void Process();
  }

  class CreditCardPayment : Payment
  {
      public override void Process() { /* ... */ }
  }
  ```

- `sealed` - тюлений класс, который нельзя наследовать.
  - Для метода модификатор `sealed override` указывает, что нельзя переопределить метод в дочерних классах
  - `sealed class` нельзя унаследовать

  ```c#
  sealed class Final : Base { }  // нельзя наследовать

  class Some : Base
  {
      public sealed override void Method() { } // нельзя переопределить в потомках
  }
  ```

- Порядок инициализации (вызова конструкторов)
  - Конструкторы вызываются от базового к производному, гарантируя, что к моменту инициализации собственных полей производного класса, родительские параметры уже инициализированы.
  - Конструкторы по умолчанию вызываются неявно
  - При отсутствии конструктора по умолчанию и наличии параметризованного конструктора, параметры в него должны быть явно переданы из конструктора производного класса через `: base(...)`

  ```c#
  class Base
  {
      public Base(int x) { /* ... */ }
  }

  class Derived : Base
  {
      public Derived(int x, int y) : base(x) { /* ... */ }
  }
  ```

- Ключевое слово `base` - служит для обращения из производных классов к членам родительского класса. Например, для вызова из производного класса реализации базового. `base` работает только в нестатических членах.
  
  ```c#
  class Derived : Base
  {
      public Derived() : base() { }

      public override void Method()
      {
          base.Method(); // вызывает реализацию в базовом классе
          // дополнительная логика
      }
  }
  ```

- Ключевое слово `this` - ссылка на экземпляр класса из которого был вызван метод. Позволяет получить доступ к членам текущего экземпляра класса из которого был вызван метод. Позволяет решать конфликт имён при совпадении имён полей с именами параметров
  
  ```c#
  class Person
  {
      private string name;
      public Person(string name)
      {
          this.name = name; // разрешает конфликт
      }
      public Person() : this("Unknown") { } // цепочка конструкторов
  }
  ```

- Сокрытие базовых членов в производных.
  
    Если класс объявляет метод с той же сигнатурой, что уже существует в родительском и не использует `override`, то происходит не замещение исходного, а сокрытие. Доступ к исходному остаётся через приведение типов. Если не использовать модификатор `new` - компилятор выдаст предупреждение. Обе реализации будут существовать параллельно, но базовая не будет доступна без приведения типа.

    ```c#
    class Base
    {
        public void Method() { }
    }

    class Derived : Base
    {
        public new void Method() { } // скрывает, а не переопределяет
    }
    ```

    `new` в отличие от `override` не требует виртуальности исходного члена, скрывает его реализацию, а не переопределяет, полиморфизм не работает (вызов по приведённому типу ссылки) и создаётся независимый член с тем же именем, не затрагивая иерархию переопределения.

- Наследование и интерфейсы
  
    Ключевой принцип, определяющий следует ли применить наследование или интерфейсы: наличие общего контекста.

    Если группа сущностей, помимо поведения, разделяют общий контекст данных и поведение, то следует применять наследование, если схоже только поведение - интерфейсы.

    Пример плохой иерархии наследования: иерархия котики-собачки-пингвины и т.п. животные. Иерархия биологическая не связана с иерархией даталогической. Для примера можно взять утку, стрижа и пингвина. Утка умеет плавать, летать, ходить по земле. Пингвин умеет плавать и ходить по земле. Стриж проводит всю жизнь в воздухе, к поверхности земли не приспособлен, плавать тоже не умеет, мало того, полётные остевые перья есть только у утки и стрижа, а пуховые перья для защиты от воды есть только у утки и пингвина. Казалось бы, все они птицы и разделяют контекст класса `Bird`, но в действительности между ними есть только общее яйцекладка и не более того. В этом случае, следует создать 3 интерфейса и обязать отдельные классы реализовать функциональность, которая относится к этим интерфейсам, а не городить наследование, поскольку нарушается принцип подстановки. Либо реализация класса птицы перестаёт уметь летать, ходить, плавать и теряется смысл в наличие базового класса.

    Пример хорошего наследования: Платёжные документы. Их много разных, но все они разделяют контекст: у всех есть номер, у всех есть дата, в каждом есть сумма. А далее различные детали: есть чек, счёт, накладная, и т.п. все эти документы имеют контекст и общее поведение в рамках платёжного документа, но разную реализацию этого поведения. Здесь имеет смысл наследование.

    ```c#
    abstract class Document
    {
        public string Id { get; set; }
        public DateTime Date { get; set; }
        public decimal TotalAmount { get; set; }

        public virtual void Validate()
        {
            if (string.IsNullOrEmpty(Id)) throw new ArgumentException("ID обязателен");
            if (TotalAmount < 0) throw new ArgumentException("Сумма не может быть отрицательной");
        }

        public abstract string GetSummary();
    }

    class Invoice : Document
    {
        public DateTime DueDate { get; set; }

        public override void Validate()
        {
            base.Validate();
            if (DueDate < Date) throw new ArgumentException("Дата оплаты не может быть раньше даты документа");
        }

        public override string GetSummary() => $"Счёт #{Id} на сумму {TotalAmount}, оплатить до {DueDate:dd.MM.yyyy}";
    }
    ```

    Ключевой принцип применения наследования: наследование применяется там, где у группы сущеностей имеется общее состояние и поведение с единым интерфейсом взаимодействия, но имеющим различную реализацию. А так же, где строго соответствует отношение `IS-A` т.е. `Платёжное поручение IS-A Документ`. Формально пингвин `IS-A` птица, но если птица определяет полёт, то `IS-A` уже не работает, ведь пингвин не летает и требуется городить зоопарк классов типа "летающая птица", "не летающая птица", "плавающая не летающая птица", "плавающая летающая птица". А это плохо.

    Ключевой принцип применения интерфейсов: группа сущеностей имеет единое поведение и интерфейс взаимодействия, но различный контекст и реализацию.
